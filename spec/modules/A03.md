## Goal
Real-time push channel from API to each logged-in agent's UI: agent state, current call info, lead-changed, supervisor pings, system messages. Subscribes to Redis pub/sub `t:{tid}:broadcast:agent:{user_id}` and forwards to the WS client.

## Dependencies
- F05 (auth), T01 (events flow into Redis), F04 (pub/sub)

## Blocks
- A04, A05, A06, A07 (UI updates)
- S01 (wallboard fan-out shares the gateway)

## Public interface
**WS endpoint:** `wss://api.example.com/api/ws` (or `/api/agent/ws`)
**Auth:** JWT in initial connection (query param or first message).
**Server → client messages (JSON):**
```
{ type:'state', payload:{ status, since, pause_code? }}
{ type:'call', payload:{ uuid, lead, started_at, state }}
{ type:'call_ended', payload:{ uuid, hangup_cause, talk_seconds }}
{ type:'transfer_complete', payload:{...}}
{ type:'system', payload:{ level, msg }}
{ type:'sup_ping', payload:{ from }}
{ type:'ping' / type:'pong' }
```
**Client → server:**
```
{ type:'pong' }
{ type:'state_change', payload:{ status, pause_code? }}    // alternative to REST POST
```

## Verification phase
1. Agent logs in; WS connects in <1s.
2. Manually publish to `t:1:broadcast:agent:42` via redis-cli; UI receives.
3. Kill API; client reconnects within 5s with backoff.
4. Two tabs of same agent: both receive state events (no double-handling state changes).
5. Heartbeat: client sends pong; server doesn't disconnect.
6. Wrong JWT closes connection with 4001.

## Test phase
**Unit:** msg schema validation; reconnect logic.
**Integration:** real Redis + Fastify; assert delivery.
**E2E:** Playwright watches state change in UI after Redis publish.
**Run:** `npm test -- ws`

## Acceptance criteria
- [ ] WS auth via JWT.
- [ ] Heartbeat every 15s.
- [ ] Auto-reconnect with backoff (1s, 2s, 4s, ... up to 30s).
- [ ] Multi-tab safe.
- [ ] Throttling enforced.
- [ ] All msg types schema-validated.
- [ ] Message latency Redis-publish→client p99 < 100ms.

## Risks
- **Connection storms on API restart** — implement jittered reconnect.
- **Memory growth from idle connections** — close idle (>5 min no msg + no pong) connections.
