## Goal
Given an E.164 phone number, return `{ tz_name, tz_offset_min, state }` using NANP area-code lookup. Used at lead insert time and at the originate compliance gate.

## Dependencies
- F02 (`phone_codes` table, seeded)

## Blocks (downstream)
- D01 (resolves at insert)
- E01 / E02 (call-time gate)
- C01 (timezone enforcement)

## Public interface
```typescript
function resolvePhoneTz(e164: string): { tzName: string; tzOffsetMin: number; state?: string; valid: boolean }
function localTimeAt(e164: string, when?: Date): { hour: number; minute: number; tzOffsetMin: number }
function isInWindow(e164: string, start: HHMM, end: HHMM, when?: Date, stateOverrides?: StateOverrides): boolean
```

## Research phase
**MCPs:**
- `mcp__exa__web_search_exa`: "NANP area code list with timezone CSV public domain 2026"
- `mcp__exa__web_search_exa`: "DST handling timezone offset Node luxon vs date-fns-tz"

**Questions:**
- DST handling — must use IANA tz names + Luxon, not static offsets.
- US territories (PR, GU, VI, AS, MP) — known-static offsets, no DST in some.
- Mobile portability — area code may not match called party's actual location. We accept this; document as a caveat (industry standard).

## Plan phase
**Deliverable:** `spec/modules/D03/PLAN.md`:
