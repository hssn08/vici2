## Goal
JWT-based authentication for the API, role-based access control (agent / supervisor / admin / superadmin), and per-user SIP credentials issued at user creation so the browser softphone can register against FreeSWITCH.

## Dependencies
- F02 (users, user_groups tables)

## Blocks (downstream)
- A01, M01 (UI auth flows)
- A02 (browser needs SIP creds)
- T02 (carrier admin RBAC)
- All API routes

## Public interface
**REST endpoints (under `shared/openapi/openapi.yaml` namespace `auth`):**
- `POST /api/auth/login` → `{ access_token, refresh_token, sip_creds: { username, password, ws_uri }, user: { id, role, ... } }`
- `POST /api/auth/refresh` → new tokens
- `POST /api/auth/logout`
- `POST /api/auth/me` → current user
- `POST /api/auth/sip/rotate` (admin or self) → rotates SIP password
**Internal middleware:**
- `requireAuth(...roles[])` Fastify decorator
- `requireTenant()` ensures `tenant_id` in JWT matches resource
- `requireOwn(resourceUserId)` for self-only routes
**FreeSWITCH integration:**
- Per-user SIP creds stored in `users.sip_password` (encrypted at rest)
- A renderer writes them to `freeswitch/conf/directory/default/<user_id>.xml` at user create/update; ESL reload triggered

**MCPs:**

**Questions to answer:**
- JWT in Authorization header vs httpOnly cookie? Recommendation: httpOnly cookie for browser, header for API tokens.
- Refresh-token rotation strategy? Recommendation: opaque tokens stored in Redis with `t:{tid}:refresh:{token_id}` → user_id; rotate on each refresh.
- How does FreeSWITCH look up directory users? Either static XML (we generate on user create) or `mod_xml_curl` (we serve from API on demand). For Phase 1: static XML with reload on change.
- Where does the SIP password come from? Generated at user create (32-char random); stored encrypted; sent to client only at login (so browser doesn't need long-term storage).
- Which encryption library for password-at-rest? `node-forge` AES-GCM with envelope-encryption (DEK + KEK in env).
- How long should JWT TTLs be? Access 15 min, refresh 30 days, sliding window.

**Pseudocode — login:**
```typescript
async function login(req) {
  const { username, password } = req.body;
  await rateLimit(req.ip, username);
  const user = await prisma.user.findUnique({ where: { username }});
  if (!user || !await argon2.verify(user.passwordHash, password)) {
    await audit({ event: 'login.fail', tenant_id, username, ip: req.ip });
    throw new AppError('INVALID_CREDENTIALS', 401);
  }
  const sipPasswordPlain = decrypt(user.sipPasswordEncrypted, KEK);
  const accessToken = signAccess({ uid: user.id, tid: user.tenantId, role: user.role });
  const refreshTokenId = await issueRefresh(user.id);
  await audit({ event: 'login.ok', tenant_id, user_id: user.id });
  return {
    access_token: accessToken,
    refresh_token: refreshTokenId,
    user: { id, role, full_name, ... },
    sip_creds: {
      username: String(user.id),
      password: sipPasswordPlain,
      ws_uri: process.env.FS_WSS_URI,   // wss://fs.example.com:7443
      domain: process.env.FS_SIP_DOMAIN
    }
  };
}
```

**Pseudocode — FS directory user XML render:**

```xml
<include>
  <user id="${user_id}">
    <params>
      <param name="password" value="${sip_password}"/>

</include>
```

## Verification phase
**Manual checks:**
1. Create user via admin API; SIP XML appears at `freeswitch/conf/directory/default/42.xml`; ESL reload succeeds.
2. `curl -X POST /api/auth/login -d ...` returns valid response.
3. Decoded JWT has expected claims; expires in 15 min.
4. Wrong password 5x → 6th attempt rate-limited.
5. Refresh token works once; second use of same token rejected (rotation).
6. `POST /api/auth/sip/rotate` issues a new password; old one stops working at next FS reload.
7. SIP register with the issued creds against FS internal profile succeeds.
8. RBAC: agent JWT can't hit `/api/admin/*`; admin can hit everything.
9. Audit table has entries for each login.

## Risks
- **KEK rotation** — if you rotate the KEK, all SIP passwords need re-encryption + FS XML re-render. Document; defer actual rotation tooling to Phase 4.
- **FS XML reload races** with concurrent user updates — wrap in a single-flight queue.
- **Cookie vs header for browser** — pick one and stick to it. Mixing creates CSRF surface.
