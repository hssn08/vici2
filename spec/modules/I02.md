# Module I02 — DID Inbound Routing

| Field | Value |
|---|---|

## Goal
Public dialplan that, on inbound INVITE from a carrier, looks up the destination DID in `did_numbers`, routes to the configured target (ingroup, IVR, agent, voicemail). Routing decision served via `mod_xml_curl` from API.

## Dependencies
- T02 (carriers + DIDs), I01 (ingroups), I03 (IVRs), F03 (mod_xml_curl loaded — add)

## Blocks
- (none new; completes inbound)

## Public interface
- API endpoint: `POST /api/internal/fs/dialplan` — FS xml_curl source
- Dialplan in `public` context routes to `default` after tagging

## Verification phase
1. DID `+12125550100` → ingroup "SUPPORT"; inbound call routes there.
2. DID → IVR; inbound call hears menu.
3. DID → direct agent (extension 1042); rings agent.
4. DID → voicemail.
5. Unknown DID → drops with 404-equivalent.

## Test phase
**Integration with SIPp** simulating each DID type.

## Acceptance criteria
- [ ] All 4 destination kinds work.
- [ ] xml_curl response in < 100ms p95.
- [ ] Cache hit > 95%.
- [ ] Audit log per inbound call.

## Risks
- **xml_curl latency** delays call setup. Aggressive caching.
