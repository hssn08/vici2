## Goal
Bootstrap Next.js 14 app router with Tailwind, Zustand, login flow, protected route groups (`(agent)`, `(admin)`, `(sup)`), session refresh, and the foundation other UI modules build on.

## Dependencies
- F05 (auth API)

## Blocks (downstream)
- All A* and M* and S* modules

## Public interface
- `web/` Next.js app
- Login page at `/`
- Agent shell at `/(agent)/dashboard` (placeholder)
- Admin shell at `/(admin)`
- Supervisor shell at `/(sup)`
- `useAuthStore` (Zustand) — exposes `user`, `accessToken`, `sipCreds`, `login()`, `logout()`, `refresh()`
- `apiClient` (typed via openapi-typescript) — auto-injects auth header

## Verification phase
1. `cd web && npm run dev` opens at localhost:4000; login page renders.
2. Login with admin creds; redirected to `/(admin)`.
3. Login with agent creds; redirected to `/(agent)/dashboard`.
4. Wrong role accessing `/(admin)` → 403 / redirect to dashboard.
5. Refresh page; session persists (cookie).
6. Logout clears session; redirects to login.
7. API call after token expiry triggers refresh transparently.
8. Mobile responsive at viewports 375 / 768 / 1280.

## Test phase
**Unit (vitest):** Zustand store, auth flows.
**E2E (Playwright):** login, route-protection, refresh, logout.
**Run:** `cd web && npm test && npm run test:e2e -- auth`

## Acceptance criteria
- [ ] Three role-segregated route groups.
- [ ] Auth flow complete (login → use → refresh → logout).
- [ ] httpOnly cookie strategy.
- [ ] Typed API client from OpenAPI.
- [ ] Tailwind + base tokens configured.
- [ ] Mobile responsive shell.
- [ ] Lighthouse score >90 on the empty shell.

## Risks
- **OpenAPI not yet stable** — generate types from a stub for now; regenerate during PRs.
- **Refresh-token race** in concurrent requests — debounce or use a refresh-promise pattern.
