## 1. Executive summary (10 bullets)

1. **T04 is a thin policy wrapper.** Per T01 PLAN §16.2 the T01↔T04 boundary is now FROZEN: T01 owns the ESL transport (`*esl.Client.Originate(OriginateRequest) (callUUID, error)`), reconnect, circuit breaker, defense-in-depth rate limiter, and BACKGROUND_JOB correlation; T04 owns the **compliance gates + audit insert + high-level intent → transport translation**. T04 imports T01; T01 must never import T04. The whole T04 implementation is roughly **400 LOC of orchestration + 300 LOC of tests**, not a re-implementation of originate.

2. **Five pre-originate checks, hard-ordered cheap-to-expensive, fail-fast.** (1) Concurrent-call cap per gateway (T02 §9 — Valkey counter, ~150 µs); (2) Drop-rate safe-harbor cap from E05 (Phase 2; Valkey gauge, ~150 µs; Phase 1 returns ALLOW); (3) TCPA window via C01.Check (~1 ms cached; defense-in-depth — E01 hopper filler already gated, but lead may have been claimed >5 min ago); (4) DNC final-scrub via D05 Bloom + MySQL fallback (~0.7 ms p99 hot, 1.2 ms with one false positive); (5) Recording-consent state policy via C02.CheckConsent (~200 µs in-process — pure data lookup). Each check that BLOCKS short-circuits and writes one `originate_audit` row tagged with the failing gate; the originate is **not** sent to T01.

3. **Originate-then-park is the universal target for Phase 1 + Phase 2.** Per T01 PLAN §7.3, T01 already issues `bgapi originate {…vars…}sofia/gateway/<gw>/<num> &park()` in **all** modes. The differentiator between modes is the value of `execute_on_answer`: PROGRESSIVE/MANUAL/PREVIEW set `execute_on_answer=transfer:conference_<aid> XML default` so the FS engine atomically transfers on CHANNEL_ANSWER; PREDICTIVE leaves `execute_on_answer=` empty and the answer event triggers an explicit `T01.UUIDTransfer` from the picker (E04) once it has selected an idle agent. **Recommendation: park-then-bridge for both,** because (a) it lets the C01 TCPA gate re-evaluate after answer (the boundary check at 8:59 PM that T04.md §3 demands), (b) it gives R01/C02 a hook for the recording-consent prompt before the agent is bridged (mandatory in the 12 two-party-consent states, see §9), and (c) it lets E05 (drop-rate) record an abandon if no agent is available, and play the safe-harbor message before hangup. The cost — ~50 ms additional connect latency for the second `uuid_transfer` — is well below the 200 ms agent-desktop latency target we already accept.

4. **`originate_audit` is a NEW table, separate from `call_log`.** Per T01 PLAN §16.4 ("T04 is the SOLE writer of `originate_audit`") and the T04 brief, F02 PLAN must be amended (additive, no RFC) to add `originate_audit`. Schema proposal in §7. Critical properties: (a) one row per **attempt** (idempotency-keyed by `attempt_uuid`, never updated), (b) row written **before** T01.Originate is invoked so we always have evidence of intent even when T01 fails, (c) `outcome` column updated *only* once (UPDATE WHERE outcome IS NULL → SUCCESS|FAILURE — single-shot), (d) `RANGE COLUMNS(originated_at)` monthly partitioning matching the other partitioned logs (F02 PLAN §3 list).

5. **`attempt_uuid` is the idempotency key for the entire dial pipeline.** Generated by the caller (E04 picker / A04 manual-dial endpoint / N01 click-to-dial REST) at the moment of the user/system intent. Sent through gRPC to T04. T04 checks `originate_audit.attempt_uuid` UNIQUE constraint as the dedup gate — if a row exists, return the prior decision/result without re-running gates (idempotent semantics matching Stripe / Amazon Connect `DialRequest` / FreeSWITCH `origination_uuid` patterns [refs §12]). The `attempt_uuid` is also passed to T01 as the **pre-supplied** `Job-UUID` and `origination_uuid` (T01 PLAN §7.3 already supports this), giving us a single ID that ties intent → audit row → ESL job → channel UUID → `call_log` row → recording → disposition. One ID, six tables.

6. **Channel-var blob is built by T04, executed by T01.** T04 assembles a `map[string]string` of FS channel vars from the campaign + lead + carrier + recording-consent decision, then hands it to T01 inside `OriginateRequest.ChannelVars`. T01 sorts + escapes + serializes them into the `{k=v,k=v}` blob. The full per-call var schema is in §4: a 16-key matrix covering caller-ID, originate behaviour, correlation IDs, recording, branded-calling SIP X-headers, and the consent-decision flag that R01's dialplan extension consults to decide whether to play the beep tone and start `record_session` before bridging. Full list in §4.

7. **Caller-ID strategy is a 4-tier waterfall.** (1) Per-call override (e.g., A07 transfer module sets a fixed CID for the closer leg) > (2) Per-list override (`lists.caller_id_override` from F02 — already exists; not yet a column? — file F02 amendment if missing) > (3) Local-presence pool (X05 Phase 3.5: pick a DID whose NPA-NXX matches lead.NPA-NXX from `did_numbers` pool) > (4) Campaign default (`campaigns.caller_id_override` from F02 PLAN §4.6). Phase-1 wires up tiers (1) and (4); X05 fills (3) later. Branded-calling X-Brand SIP header (N05 Phase 4) is set as a separate channel var and is independent of the CID source. **Recommendation:** keep T04's caller-ID logic in `dialer/internal/originate/cid_picker.go` as a single function returning `(e164, name string, source enum)` so the branded-calling module + local-presence module can override behavior cleanly later [refs §12].

8. **Failure handling is a 5-state matrix, all observable.** Compliance-fail → audit row with `outcome` ∈ {`TCPA_BLOCKED`, `DNC_BLOCKED`, `CONSENT_BLOCKED`, `GATEWAY_LIMIT`, `RATE_LIMITED`}, metric incremented, error returned to E04/E02; the hopper claim is **released** (lead returned to pool with status reflecting why — TCPA_OOW for window, DNC for scrub, etc., which D04 must enumerate). T01 transport-fail (`ErrCircuitOpen`, `ErrFSDead`, `ErrJobOrphaned`, `ErrRateLimited`) → audit row updated to `GATEWAY_FAIL` / `JOB_ORPHANED`, hopper claim released with `CARRIER_FAIL` (E01 will retry the lead in N minutes), metric incremented. BACKGROUND_JOB returns non-zero (carrier rejected with 4xx/5xx) → audit row updated to `GATEWAY_FAIL` with `error_message` carrying the FS hangup_cause; pacing (E03) consumes the dropped-window stream T01 already writes to (T01 PLAN §10.3) — T04 does not duplicate that write. Hard timeout > `originate_timeout` (default 30 s) without ever seeing CHANNEL_ANSWER → T01 returns success at the bgapi level but downstream HANGUP_COMPLETE carries `NO_ANSWER`; T04 keeps `outcome=SUCCESS` (the originate succeeded transport-wise) — `call_log` records the disposition. Job orphaned via E06 janitor sweep (FS process died holding our `in_flight` HASH) → janitor emits a synthetic JOB_ORPHANED CUSTOM event; T04's stream consumer updates the audit row.

9. **The whole dial is observable via 5 Prometheus metrics.** `vici2_t04_originate_total{tenant,campaign,mode,outcome}` (counter, the master KPI), `vici2_t04_compliance_blocked_total{gate}` (counter, breakdown of which gate fired), `vici2_t04_check_latency_seconds{gate}` (histogram, validates gate budgets are met), `vici2_t04_audit_insert_latency_seconds` (histogram, MySQL write latency), `vici2_t04_inflight` (gauge, in-flight Originate calls awaiting BACKGROUND_JOB → useful for capacity planning). Alerts: `compliance_blocked_total` rate spike > 5 σ over 1h (probably DNC sync broke or campaign config error), `check_latency_seconds` p99 > 50 ms (Valkey/MySQL slowness), `inflight` > 5× steady-state for >2 min (E04 picker stuck or carrier all-down).

10. **Open question for orchestrator: where does `originate_audit` row get inserted?** Two options: (a) **synchronous** — INSERT before `T01.Originate`, UPDATE outcome after BACKGROUND_JOB (one extra MySQL roundtrip per dial; durability guaranteed; matches F02 PLAN's "evidence first" stance for `audit_log` + `drop_log`); (b) **async** — write to a `t:{tid}:audit:originate` Stream, consumer in `workers/` flushes to MySQL in batches (lower latency; risk of in-flight loss on Valkey crash before flush). **Recommendation: synchronous.** TCPA evidence durability outweighs the ~1 ms saved (cf. F02 PLAN §1.5 "innodb_flush_log_at_trx_commit=1 NOT NEGOTIABLE — telephony writes are TCPA evidence"). For a 100-CPS dialer, 100 inserts/s on the partitioned table is trivial — F02 sized for ~200k inserts/min on `call_log` alone. Document in PLAN.

## 2. Originate scenarios (4 modes)

The four modes from the T04 brief map to **two** wire-level dispatch patterns. The dispatch pattern (CONFERENCE vs PARK) is what changes; everything else is just policy + which `OnAnswerAction` we choose at originate time.

### 2.1 PROGRESSIVE (1:1 — agent selected pre-dial)

| Stage | Action |
|---|---|
| Trigger | E04 picker has READY agent X; E02 pacing requests one originate. |
| `attempt_uuid` source | E02 generates `uuidv4` at "I'm about to dial for agent X" decision. |
| Compliance gates | Run all 5 (concurrent-cap, drop-cap, TCPA, DNC, consent). |
| `Mode` | PROGRESSIVE |
| `DialTarget` | CONFERENCE |
| T01 OriginateRequest | `OnAnswer = OnAnswerConference{Name: fmt.Sprintf("conference_%d", agentX)}` |
| Wire form | `bgapi originate {execute_on_answer=transfer:conference_X XML default,...}sofia/gateway/<gw>/<num> &park()` |
| Why park-first then transfer | T01 already writes the universal `&park()` ending. `execute_on_answer` is the FS-side trigger that fires the transfer **after** park settles. Idempotent from FS's perspective; gives R01 dialplan a CHANNEL_ANSWER hook to start recording before the conference mux. |
| On answer | FS auto-bridges customer leg into agent X's conference. T04 plays no further role. |
| Failure (no answer / busy / reject) | T01's BACKGROUND_JOB callback returns the FS originate-disposition string; T04 updates audit row and lets the lead go back to the hopper claim release path with the appropriate D04 status (NA / B / DC). |

### 2.2 PREDICTIVE (N:1 — agent assigned post-answer)

| Stage | Action |
|---|---|
| Trigger | E02 pacing decides to overdial (current dial-level > ready-agents). |
| `attempt_uuid` source | E02 generates one per outbound attempt (multiple per agent slot). |
| Compliance gates | Same 5. |
| `Mode` | PREDICTIVE |
| `DialTarget` | PARK |
| T01 OriginateRequest | `OnAnswer = OnAnswerPark{}` |
| Wire form | `bgapi originate {…,execute_on_answer=park}sofia/gateway/<gw>/<num> &park()` (`park` redundant with trailing `&park()` but explicit; harmless) |
| On answer (CHANNEL_ANSWER event) | E04 picker subscribes to `events:vici2.call.answered`. On event: (a) re-run TCPA gate (the boundary check), (b) atomically claim a READY agent from `t:{tid}:agents:ready` Valkey ZSET, (c) call `T01.UUIDTransfer(callUUID, "conference:conference_<aid>@default")`. If no agent claimable within `safe_harbor_window_ms` (default 1800 ms — well under the FCC 2-second rule [refs §12]): T04 → `T01.UUIDPlayAndHangup(callUUID, campaign.safe_harbor_audio)`, write `drop_log` row (E05 territory), update `originate_audit.outcome = SUCCESS` (the originate succeeded; the abandon is a separate fact). |
| Why park-then-transfer | Standard FS pattern (cited in [Brian West, FS-users 2010]; [SignalWire docs §IVR Originate]; [switchio docs §Call generation]). Park gives the dialer an explicit "I have time to find an agent" window. |
| Failure | Same as PROGRESSIVE. |

### 2.3 MANUAL (agent presses dial in UI)

Identical to PROGRESSIVE except:
- `attempt_uuid` is generated by A04 (manual-dial UI) or its API endpoint.
- C01 also runs an **upstream** UI-side check (A04 PLAN responsibility) so the agent never sees a "go" button for an OOW lead. T04's check is final defense.
- `agent_id` is always set (the agent who clicked).

### 2.4 PREVIEW (agent reviews lead, then confirms)

Identical to MANUAL except:
- Two-step flow: (a) agent UI fetches lead → (b) UI shows preview screen → (c) agent clicks "Dial".
- The TCPA gate is run on lead-fetch *and* on dial; if the window closed during the preview interval, the dial button greys out.
- T04 sees a normal MANUAL-shaped originate at "Dial" time. The PREVIEW state is entirely in A04/M02.

### 2.5 Mode → DialTarget recommendation

| Mode | DialTarget | Rationale |
|---|---|---|
| PROGRESSIVE | CONFERENCE | Agent is dedicated; latency-optimal. |
| PREDICTIVE | PARK | Mandatory — agent not yet picked. |
| MANUAL | CONFERENCE | Agent is the originator. |
| PREVIEW | CONFERENCE | Agent is the originator. |

The `DialTarget` enum lives in T04's public API to keep the call sites self-documenting; T04 maps it to the right `OnAnswerAction` internally.

## 3. Pre-originate check pipeline (order matters)

Order is **cheap → expensive → most-likely-to-fail-fast**. Each check that returns BLOCK short-circuits the rest, writes an `originate_audit` row, returns to caller. Per-check budget under §1 #2.

### 3.1 Step 1 — Concurrent-call cap per gateway (T02)

- **Read:** Valkey counter `t:{tid}:gw:{gateway_id}:active` (T02 RESEARCH §9.2).
- **Compare to:** `gateways.max_concurrent` (cached in process; reload on T02 update event).
- **BLOCK reason:** `GATEWAY_LIMIT`.
- **On block:** the lead is **not** wasted — T04 returns `ErrGatewayLimit` and E02 (pacing) decides whether to wait or pick a sibling gateway (T01 PLAN §9 multi-FS dispatch is sibling-gateway equivalent at the carrier-row layer).
- **Why first:** ~150 µs check; very common to hit when a single carrier is saturated — fast rejection avoids the more expensive checks.

### 3.2 Step 2 — Drop-rate safe-harbor cap (E05; Phase 2)

- **Read:** Valkey gauge `t:{tid}:campaign:{cid}:drop_rate_30d` (E05 territory; computed by E03 from the F04-frozen `t:{tid}:campaign:{cid}:drop_window` Stream that T01 already writes to per T01 PLAN §10.3).
- **Compare to:** `campaigns.adaptive_drop_pct` (F02 PLAN §4.6 — default 1.5%).
- **BLOCK reason:** `RATE_LIMITED` (subreason `DROP_CAP`).
- **Phase 1 behavior:** stub returns ALLOW (no E03 yet).
- **Why second:** still a Valkey-only check; ~150 µs.

### 3.3 Step 3 — TCPA window check (C01.Check)

- **Read:** call C01's Node-side `assertCallWindow({phoneE164, campaignId, leadId, when=now(), enforcementPoint='originate'})` via the gRPC stub C01 PLAN §10 will define (`shared/proto/compliance.proto` `CallWindowService.Check`).
- **Cache:** C01 has a 60s in-process cache keyed `(phone, floor(when/300))` per its RESEARCH §11 #8. T04 reads through the cache.
- **BLOCK reason:** `TCPA_BLOCKED` (sub: `OUTSIDE_WINDOW` / `STATE_BLACKOUT` / `HOLIDAY` / `UNKNOWN_TZ_DENY`).
- **Why third:** ~1 ms cached, 5 ms cold (gRPC + Luxon arithmetic). Defense-in-depth — E01 hopper filler already ran this check at hopper-fill time, but per C01 RESEARCH §7.2 "time elapsed between hopper insert and dial can cross the 9pm boundary on busy systems" — this is the **regulatory** check, the hopper one is just an optimization.
- **Lead status update on block:** D04 must define a TCPA-block status (e.g., `TCPA` or reuse `XDROP`) — surface as open question in §11.

### 3.4 Step 4 — DNC final scrub (D05.Check)

- **Read:** D05's Go SDK `dnc.IsDnc(ctx, lead.tenant_id, phone, lead.state, campaignID)` per D05 RESEARCH §4.1.
- **Sources checked (all of):** `internal`, `state` (matched by lead.state), `federal`, `litigator` (Phase 2). The check returns `(dnc bool, sources []string)`.
- **Bypass token:** if caller passed `req.BypassToken != ""`, T04 atomically `SET NX EX 60` redeems the Valkey token (D05 RESEARCH §7.2). Bypass writes a chained C03 audit_log entry; never silent.
- **BLOCK reason:** `DNC_BLOCKED` (sub: which source).
- **Why fourth:** ~0.7 ms hot (Bloom hits all "negative"), ~1.2 ms with one false positive that requires MySQL confirm, max ~5 ms over the four sources. Cheaper than C01 hot, more expensive cold; placement after C01 reflects the relative likelihood of a hot-cache TCPA-fail (very common at 8 PM transitions) vs DNC false-positive (rare).
- **Lead status on block:** D04 already has `DNC` status.

### 3.5 Step 5 — Recording-consent state policy (C02)

- **Input:** lead.state (or lead.known_timezone-derived state if state is null).
- **Output:** `consent_decision ∈ {ALLOW, PROMPT, SKIP_RECORDING}` (where PROMPT means R01's dialplan must play "this call may be recorded" beep + listen for opt-out DTMF before bridging; SKIP_RECORDING means the recording channel-var must be off entirely).
- **State matrix (12 two-party-consent states as of 2026 [refs §12]):** CA, CT, DE, FL, IL, MD, MA, MI, MT, NH, OR, PA, WA. Plus all **federal calls are 1-party** by default (FL, IL, MA, PA, WA are stricter; the others vary). T04 looks at the **called-party state** (lead.state) — if the call is interstate and either party's state is 2-party, conservative posture is to PROMPT.
- **Phase 1 implementation:** since C02 has not yet started research, T04 ships with a vendored copy of the state matrix in `dialer/internal/originate/consent.go` and a stub `func CheckConsent(state string, callerState string) ConsentDecision`. When C02 lands (Phase 1 still — same phase), T04 swaps the stub for the C02 import. This is documented as **the C02-dependency assumption** and tracked in §11.
- **Recommendation:** in PROMPT mode, set channel var `vici2_consent_required=true` and `vici2_consent_audio=campaigns.consent_prompt_audio`; R01's dialplan extension on CHANNEL_ANSWER plays the audio + listens for opt-out DTMF, then triggers `record_session` (or skips it). The bridge to agent conference happens *after* the consent flow finishes (or immediately if state is 1-party). This adds ~2-4 s to connect time in 2-party states, which is the legal cost of doing business — document in HANDOFF.
- **Why fifth:** cheapest of all (in-process map lookup, ~200 ns) — but logically last, because it's the only check whose result we **carry forward** as a channel var rather than just a yes/no. Doing it last lets us avoid setting the var on calls that BLOCK earlier.

### 3.6 Step 6 — T01.Originate

If all 5 gates pass, T04 inserts the `originate_audit` row, then calls `T01.Client.Originate(ctx, transportReq)`. T01's own rate limiter (T01 PLAN §13 — per-FS + per-gateway token bucket, defense-in-depth) is the **last** gate — it fires only if E02 pacing has gone bananas. If T01 returns `ErrRateLimited`, T04 updates the audit row to `RATE_LIMITED` (sub: `T01_BUCKET`) and returns to caller.

## 4. Channel-var schema (FS originate args set by T04)

This is what T04 puts into `OriginateRequest.ChannelVars` for T01 to splice into the `{k=v,…}` blob. The 16 keys split into 5 groups.

### 4.1 Group A — Caller-ID (driven by §5 picker)

| Var | Source | Notes |
|---|---|---|
| `origination_caller_id_number` | §5 waterfall | E.164 with `+` (carriers like Twilio require `+`; Telnyx/Bandwidth accept either; the FS originate doc treats `+` as significant — set always) |
| `origination_caller_id_name` | §5 waterfall | UTF-8 max 64 chars; carrier may strip |
| `effective_caller_id_number` | same as origination_caller_id_number | belt-and-suspenders for FS-internal CID display |
| `sip_from_user` | same | survives Twilio/Bandwidth From-header rewrites per T02 RESEARCH §2 |

### 4.2 Group B — Originate behavior

| Var | Default | Notes |
|---|---|---|
| `ignore_early_media` | `true` | mandatory per T04.md "spuriously triggering answer detection"; T01 PLAN §7.3 already sets this |
| `originate_timeout` | `campaigns.dial_timeout_sec` (default 22) | ring time |
| `call_timeout` | same as originate_timeout | belt-and-suspenders for bridge variants |
| `hangup_after_bridge` | `true` for CONFERENCE; `false` for PARK | PARK keeps the channel after the agent hangs up so the dialer can `uuid_park` to the next destination if needed |
| `execute_on_answer` | `transfer:conference_<aid> XML default` for PROGRESSIVE/MANUAL/PREVIEW; `park` for PREDICTIVE | T01's `OnAnswerAction` interface in T01 PLAN §7.1 emits this |

### 4.3 Group C — Correlation IDs (round-trip on every CHANNEL_* event)

| Var | Source |
|---|---|
| `vici2_tenant_id` | `req.TenantID` |
| `vici2_lead_id` | `req.LeadID` |
| `vici2_campaign_id` | `req.CampaignID` |
| `vici2_agent_id` | `req.AgentID` (0 for predictive) |
| `vici2_attempt_uuid` | `req.AttemptUUID` |
| `vici2_carrier_id` | resolved from `req.CarrierID` |
| `vici2_gateway_id` | resolved from carrier/gateway pick |
| `origination_uuid` | = `req.AttemptUUID` (single ID across all systems — see §5 of this RESEARCH) |

T01 reads these as `variable_*` headers off the wire and uses them in event enrichment (T01 PLAN §10.1 / §11). The `vici2_*` prefix prevents name-collisions with FS's standard channel vars.

### 4.4 Group D — Recording / consent (for R01 + C02)

| Var | Source / value |
|---|---|
| `vici2_consent_required` | `"true"` if §3.5 returned PROMPT, else `"false"` |
| `vici2_consent_state` | the called-party state (`CA`, `FL`, etc.) — lets R01's dialplan log which state's policy applied |
| `vici2_recording_mode` | maps `campaigns.recording_mode` → `OFF` / `ON` / `FORCED`; R01 dialplan reads this to decide whether to call `record_session` |
| `RECORD_STEREO` | `"true"` always — R01 sec_research recommends stereo for downstream transcription quality |

**No** dialplan-level `record_session` is invoked by T04 directly. T04 just sets the vars; R01's `00_outbound.xml` extension on CHANNEL_ANSWER reads them and acts. Keeps recording policy in one module.

### 4.5 Group E — SIP X-headers (carrier-specific + branded calling)

| Var | Source |
|---|---|
| `sip_h_X-Vici2-Lead` | `req.LeadID` (downstream Salesforce CTI etc. lookups) |
| `sip_h_X-Vici2-Campaign` | `req.CampaignID` |
| `sip_h_X-Vici2-Attempt` | `req.AttemptUUID` |
| `sip_h_X-Brand` | (Phase 4, N05) branded-calling token from `did_numbers.brand_token` |
| `sip_h_X-Account-Id` | (Bandwidth-only, conditional) per T02 RESEARCH §2 if E911 carrier |

X-headers are SIP-stripped by some carriers — known issue per T02 RESEARCH §2 (Twilio strips, Telnyx and Bandwidth pass through). Document in HANDOFF.

## 5. Caller-ID strategy (4-tier waterfall + branded tokens)

### 5.1 Picker function

```
func PickCallerID(req LeadOriginateRequest, lead Lead, campaign Campaign, list List) (number, name, source) {
  // Tier 1: per-call override (highest priority)
  if req.CallerIDOverride != "" { return req.CallerIDOverride, req.CallerIDName, "per_call" }

  // Tier 2: per-list override (e.g., West Coast list uses 415 area code)
  if list.CallerIDOverride != "" { return list.CallerIDOverride, list.CallerIDName, "per_list" }

  // Tier 3: local-presence (Phase 3.5, X05) — match NPA-NXX of lead
  if campaign.LocalPresenceEnabled {
    if did := X05.PickLocalDID(lead.PhoneE164); did != nil {
      return did.Number, did.BrandName, "local_presence"
    }
  }

  // Tier 4: campaign default
  return campaign.CallerIDOverride, campaign.CallerIDName, "campaign_default"
}
```

Phase 1 wires tiers (1) and (4); X05 fills (3) later; per-list override needs F02 amendment (`lists.caller_id_override VARCHAR(16) NULL`, `lists.caller_id_name VARCHAR(32) NULL`).

### 5.2 Local-presence (X05, Phase 3.5)

Maintains a pool of DIDs per NPA-NXX in `did_numbers` (F02 PLAN §4.22). T04's picker calls X05's `PickLocalDID(leadPhone)` which:
1. Computes lead's NPA-NXX.
2. Queries Valkey set `t:{tid}:did_pool:npa_nxx:<npanxx>` for available DIDs (DIDs are rotated to avoid spam-flag accumulation).
3. Returns one with `last_used_at` > `cooldown_seconds` ago.
4. If pool empty, falls through to campaign default with metric `vici2_t04_local_presence_miss_total{npa_nxx}`.

**Per-DID call counts** (X05 PLAN territory): rotation policy — no DID used more than N times/day to avoid CNAM degradation. Out of T04 scope; T04 just calls the picker.

### 5.3 Branded calling (N05, Phase 4)

Per T02 RESEARCH §10 #5, branded calling (Twilio Branded Calling, Hiya Connect, First Orion) is **per-DID**. Once a DID is enrolled in a branded-calling program, the carrier signs the outbound call with STIR/SHAKEN A-attestation + emits a brand bundle [refs §12]. T04's only job: when the picked DID has `branded_token IS NOT NULL`, set `sip_h_X-Brand=<token>` channel var. The carrier (or downstream Tiltx-style signer per T02 RESEARCH §10 #3) emits the proper Identity header. T04 does **not** generate STIR/SHAKEN tokens.

### 5.4 Caller-ID name (CNAM)

Some carriers (Bandwidth) accept arbitrary `From-display-name`; others (Twilio) ignore it (CNAM is dipped at the terminating carrier from a dipped database). Set `origination_caller_id_name` always; document per-carrier behavior in HANDOFF.

### 7.2 Properties

- **Partition column** is `originated_at` matching F02 PLAN's pattern for `call_log` / `audit_log` / `drop_log`.
- **Composite PK** `(id, originated_at)` (partition column required in PK per MySQL).
- **`UNIQUE (attempt_uuid)`** is the idempotency mechanism — see §8.
- **No FKs** (partitioned table; F02 PLAN §3 says no FKs on partitioned tables; app-level integrity).
- **Append-mostly with one UPDATE** — INSERT happens at `originated_at`; one UPDATE at `outcome_at` flipping `outcome` from `OTHER` (provisional) to the final value. Audit immutability is **stronger** than `audit_log` here because the sole post-INSERT UPDATE is a deterministic finalize; consider a CHECK constraint that `outcome != 'OTHER'` is allowed only after `outcome_at IS NOT NULL`. But because we want the row INSERTable with `outcome='OTHER'` initially, we keep it loose.
- **GRANT model:** unlike `audit_log` which is INSERT-only, this table is INSERT + UPDATE-with-conditions. Recommendation: F02 amendment gives `vici2_app` `INSERT, UPDATE, SELECT` on this table; runtime DAL guards UPDATE to `WHERE outcome='OTHER' AND outcome_at IS NULL` so no row is ever overwritten after finalization.
- **Retention:** 7 years (TCPA), partitioning monthly, archive to S3 with object-lock per C03 after 90d (mirrors `audit_log` policy).

### 7.3 Volume estimate

100-CPS dialer at full bore: 100 originates/s × 86400 s/day = 8.64M rows/day = ~260M/month per partition. Comfortably handled by `RANGE COLUMNS(originated_at)` monthly partitioning at the F02-pinned `innodb_buffer_pool_size=16G` for dev / similar for prod. Index size on the 4 secondary indexes ~25% of row size; total disk ~70 GB/month at 100-CPS. Plan for tiered storage.

### 7.4 Why separate from `call_log`

- `call_log` rows exist only for originates that **succeed at the transport layer** (T01 wrote a `CHANNEL_CREATE` row for them). Originate attempts that fail compliance never produce a `call_log` row.
- TCPA defense package wants **every attempt** including blocked ones; folding into `call_log` would mean filling that table with thousands of "never-dialed" rows that pollute reporting queries.
- T01 PLAN §16.4 ownership split makes `call_log` write-owned by T01 (CDR fields) + api (dispo fields); pushing T04 fields into `call_log` would create a 3rd writer and contention.
- Schema evolution: `originate_audit` columns serve forensic / TCPA-defense reads; `call_log` columns serve operational reporting. Splitting tables lets each evolve without index pressure on the other.

## 8. Idempotency via `attempt_uuid`

### 8.1 Why

- E02 pacing tick may double-fire under network jitter (both pacing replicas decide to originate the same lead); the dedup gate prevents double-dial.
- A04 manual-dial UI may resubmit (browser retry on 502); dedup returns the prior decision.
- N01 click-to-dial REST callers (Salesforce CTI etc.) are required by integration norm to send an idempotency key.
- If T04 itself crashes between INSERT and T01.Originate, the `attempt_uuid` row is in MySQL with `outcome='OTHER'`; on restart, an E06 janitor sweep finds it and either re-issues T01.Originate (if `Job-UUID` was never registered) or marks it `OTHER → JOB_ORPHANED`.

### 8.2 Where the UUID comes from (one ID, six tables)

| ID role | Table.column |
|---|---|
| Idempotency key | `originate_audit.attempt_uuid` (UNIQUE) |
| Pre-supplied FS Job-UUID | T01 PLAN §7.3 — `Job-UUID: <uuid>` header |
| Pre-supplied FS channel UUID | `origination_uuid=<uuid>` channel var |
| Live state hash key | `t:{tid}:in_flight:{call_uuid}` (T01 PLAN §11.2) |
| CDR row | `call_log.uuid` (F02 PLAN §4.24 — UNIQUE on `(uuid, call_started)`) |
| Recording row | `recording_log.uuid` (F02 PLAN §4.26) |

By policy, **all six are the same UUIDv4 string** (lowercase, hyphenated, 36 chars; fits `VARCHAR(40)` everywhere). Generated **once** by the caller, propagated through the pipeline. This was suggested in T01 PLAN §16.3 sequence diagram and is now codified here.

### 8.3 Dedup algorithm

```
On T04.Originate(req):
  // Try to claim the idempotency slot atomically
  INSERT INTO originate_audit (..., attempt_uuid=?, outcome='OTHER', originated_at=NOW(6)) ...;
  // If duplicate-key (attempt_uuid already present):
  //   SELECT outcome, call_uuid FROM originate_audit WHERE attempt_uuid=?;
  //   If outcome != 'OTHER': return prior result (true idempotent replay).
  //   If outcome == 'OTHER': another T04 worker is in-flight with the same key.
  //     → return ErrInProgress so caller can poll/wait.
  // Else: continue with the gates + T01.Originate.
```

The MySQL `INSERT … ON DUPLICATE KEY UPDATE` pattern is **not** used because we don't want any UPDATE to silently change an in-flight row.

### 8.4 Naming convention

- Generated by: caller (E02/A04/N01).
- Validity: 24h (matches T01 in-flight HASH expiry per T01 PLAN §11.2).
- Replay window: T04 looks up `attempt_uuid` only against the active partition + previous month. Older attempts assumed expired; collision impossible for UUIDv4.

## 10. Hand-off to E04 (picker), E02 (pacing), E06 (janitor)

### 10.1 To E04 (agent picker, Phase 2)

- E04 calls `T04.Originate(req)` directly with `req.AgentID = pickedAgent.ID` for PROGRESSIVE; with `req.AgentID = 0` for PREDICTIVE.
- For PREDICTIVE, E04 also subscribes to `events:vici2.call.answered` consumer-group `picker`; on event, claims an agent + calls `T01.UUIDTransfer`. (T01 PLAN §17.5 already permits direct T01 calls for E06-style janitor; E04 needs the same exemption — flag for orchestrator: should E04 go through T04 even for the post-answer transfer? **Recommendation: NO** — the compliance gates were already enforced at originate; the post-answer transfer is just routing. E04 calls `T01.UUIDTransfer` directly. Document in T01 PLAN addendum.)

### 10.2 To E02 (pacing loop)

- E02 calls `T04.Originate(req)` for every dial slot it decides to fill. T04's failures are E02's signal to back off (concurrent_cap = need a sibling gateway; tcpa_blocked = lead skipped, try another lead; carrier_fail = circuit-breaker the carrier in pacing).
- E02 generates `attempt_uuid` per dial.

### 10.3 To E06 (channel/conference janitor)

- E06 sweeps `originate_audit WHERE outcome='OTHER' AND originated_at < NOW() - INTERVAL 5 MINUTE` (i.e., orphaned originates that never finalized).
- For each: cross-check FS via T01's `*esl.Client.Reconcile` (T01 PLAN §14.1) to see if a channel exists; if not → UPDATE outcome=`JOB_ORPHANED`; if yes → wait one more sweep.

### 10.4 To A04 (manual dial)

- A04's API endpoint generates `attempt_uuid`, calls T04 via gRPC.
- On `TCPA_BLOCKED` etc., A04 surfaces the reason to the agent UI ("Outside calling hours for this number").
- Manual dial is the only mode where the agent sees the BLOCK reason directly; for PROGRESSIVE/PREDICTIVE the lead just goes back to the hopper.

### 10.5 To N01 (external REST API, Phase 4)

- N01 is the click-to-dial entry for Salesforce/HubSpot. It calls T04 over gRPC; T04 treats it identically to MANUAL.
- N01 callers MUST send `Idempotency-Key` HTTP header which is the `attempt_uuid`.
