## Goal
Background loop (per active campaign, every 30s) that fills `t:{tid}:campaign:{cid}:hopper` Redis sorted-set from `leads` table after applying every filter: campaign dial_status_filter, time-zone window, DNC (federal+state+internal), called_count limit, recycle delay, lead_filter SQL, anyone-callbacks-due. Mirrors Vicidial AST_VDhopper.pl semantics.

## Dependencies

## Blocks

## Public interface
- `Engine.fillHopper(campaignID)` callable manually + on cron
- Redis key: `t:{tid}:campaign:{cid}:hopper` (ZSET)

**Pseudocode:**
```go
func (h *Filler) Fill(ctx context.Context, c Campaign) (int, error) {
    target := computeTarget(c)
    have := redis.ZCard(hopperKey(c))
    need := target - have
    if need <= 0 { return 0, nil }
    rows, err := h.db.QueryLeadsForHopper(ctx, c, need)
    if err != nil { return 0, err }
    pipe := redis.Pipeline()
    inserted := 0
    for _, r := range rows {
        // re-check TZ in case of clock drift
        if !c.calltimes.InWindow(r.PhoneE164, time.Now()) { continue }
        score := float64(r.Rank)*1e10 + float64(time.Now().UnixMicro())
        pipe.ZAdd(hopperKey(c), Z{Score: score, Member: r.LeadID})
        inserted++
    }
    pipe.Exec()
    metrics.HopperInserted.Add(float64(inserted))
    return inserted, nil
}
```

## Verification phase
1. With seeded leads + a campaign in 9am-9pm EST, hopper fills at 9:00 AM EST and stops filling at 9:00 PM EST.
2. Lead with status=NA + recycle_delay=2h not in hopper until 2h after last call.
3. Lead in DNC (federal stub) excluded.
4. Adding 1000 NEW leads → hopper repopulates within 30s.
5. Lead_filter "state IN ('CA','OR','WA')" honored.
6. ANYONE callback due → inserted into hopper with priority bump.

- [ ] Hopper sizing formula correct.
- [ ] Lead_filter SQL bound safely (no injection).

## Risks
- **Slow query at 10M+ leads** — partitioning of leads not in scope (yet); add covering indexes.
- **Lead_filter SQL injection** — use parameterized fragments only; reject DDL keywords.
