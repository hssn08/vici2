## Goal
REST + service-layer CRUD for `leads` table with search, filter, status update, custom-data validation, and bulk operations. Powers admin UI, agent lookup, and dialer engine reads.

## Dependencies
- F02 (`leads`, `lists`, `statuses`)
- F05 (RBAC; agent vs admin scope)
- D03 (timezone resolution at insert)
- D04 (status validation at update)

## Blocks (downstream)
- D02 (CSV import calls our service)
- A04, A05, A06 (manual dial / live panel / dispo)
- D06 (callbacks reference leads)
- E01 (hopper filler reads leads)
- M03 (admin lead UI)
- N01 (external add_lead API)

## Public interface
**REST (under `/api/leads`):**
- `GET /api/leads?listId=&status=&search=&page=` — paginated; admin scope
- `GET /api/leads/:id` — single lead w/ history (call_log, agent_log)
- `POST /api/leads` — create
- `PATCH /api/leads/:id` — update non-status fields
- `POST /api/leads/:id/status` — change status (validates against campaign statuses; emits `lead.status_changed`)
- `POST /api/leads/bulk-status` — `{ ids[], status, comment }`
- `POST /api/leads/:id/recycle` — re-add to dialable pool (called by D06 callbacks)
- `GET /api/leads/lookup?phoneE164=` — agent quick lookup
- `DELETE /api/leads/:id` — soft-delete (sets `deleted_at`)

**Service layer (TS):**
```typescript
class LeadService {
  create(input: LeadInput): Promise<Lead>            // resolves tz_offset from phone via D03
  update(id, patch): Promise<Lead>
  setStatus(id, status, opts): Promise<Lead>          // emits event
  lookupByPhone(e164): Promise<Lead | null>
  search(filters): Promise<Paginated<Lead>>
  bulkInsert(rows: LeadInput[]): Promise<{ inserted, skipped, errors }>   // for D02
}
```

**Events emitted (Redis stream `events:vici2.lead.*`):**
- `lead.created`, `lead.updated`, `lead.status_changed`, `lead.deleted`

2. Update name/email; modify_at updated; called_count untouched.
3. Set status to SALE for an existing campaign; lead.status='SALE'; event published; agent_log row written.
4. Bulk insert 10,000 rows in <30s; duplicates within list rejected.
5. Lookup by phone returns most recent lead.

## Test phase
**Unit:** validation, normalization, status transitions allowed/denied.
**Integration:** bulk insert + dedupe; concurrent status updates resolve to last-write-wins with proper modify_at.
**Run:** `cd api && npm test -- leads`

## Acceptance criteria
- [ ] All endpoints implemented + RBAC enforced.
- [ ] Phone normalization to E.164 at write time.
- [ ] Bulk insert ≥ 10k rows in ≤ 30s.
- [ ] Custom-data validated per-list schema.
- [ ] Events emitted after commit.
- [ ] Pagination ≤ 200 rows / page.
- [ ] Soft-delete preserves history.
- [ ] No N+1 queries in list endpoint (verified by query log inspection).

## Risks
- **Custom-data schema migrations** — leads created under old schema may fail new validation. Make validation lenient on read; strict on write.
- **Concurrent status updates** — last-write-wins is OK for status; reject if `modify_at` mismatch on critical paths.
