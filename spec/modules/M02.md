## Goal
Admin CRUD for `campaigns` with all fields exposed: dial method, dial level, drop %, hopper level, dial timeout, recording mode, scripts, hot keys, pause codes config, in-group bindings, DNC sources, callback rules, lead filter SQL.

## Dependencies
- F02, F05, M01

## Blocks
- M03 (lists assigned to campaigns), E01–E03 (read campaign config)

## Public interface
**REST:**
- `GET/POST/PATCH/DELETE /api/admin/campaigns`
- `GET /api/admin/campaigns/:id/stats` — quick rollup (active agents, leads in hopper, calls today)

**UI:**
- List view with status (active/paused, agents, drop%)
- Detail/edit form grouped by tab: Dialing, Pacing, Recording, Compliance, Scripts/Forms, Callbacks, Lists, Closer-Ingroups

## Verification phase
1. Create campaign, all defaults populated; saves successfully.
2. Edit dial_method to ADAPT_TAPERED; persisted; (Phase 2) E03 picks it up within 60s.
3. Toggle active=false; pacing stops within 60s.
4. Delete (soft); campaign no longer in list; calls in flight survive.
5. Stats endpoint returns plausible numbers.

## Test phase
**Unit:** validators (e.g., adaptive_dropped_pct ≤ 3.0).
**E2E:** form CRUD flow.

## Acceptance criteria
- [ ] All campaign fields editable.
- [ ] Validation prevents nonsensical settings.
- [ ] Active toggle takes effect within 60s.
- [ ] Stats panel.
- [ ] Audit log per change.

## Risks
- **Field sprawl** — 50+ fields can be confusing. Mitigate with sensible defaults + tooltips.
