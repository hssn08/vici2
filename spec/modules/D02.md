## Goal
Asynchronous CSV ingestion: admin uploads → file lands in S3/local → background worker streams it row-by-row, normalizes phones, validates against the list's custom_data schema, writes via D01 `bulkInsert`, reports progress and errors back to UI.

## Dependencies
- D01 (LeadService.bulkInsert)
- F02 (`lists.custom_field_schema`)
- F04 (Redis for progress reporting + job queue)

## Blocks (downstream)
- M03 (admin UI uses upload flow)

## Public interface
**REST:**
- `POST /api/admin/lists/:listId/imports` — upload CSV (multipart); returns `{ import_id }`
- `GET /api/admin/imports/:id` — `{ status: queued|running|done|failed, progress: {processed, total}, errors: [...], summary: {inserted, skipped, errors_count} }`
- `GET /api/admin/imports/:id/errors.csv` — error report download

**Worker:**
- Listens on Redis queue `vici2:queue:lead-import`; processes one job at a time per worker.
- Streams CSV (does not load into memory); chunks 500 rows per `bulkInsert` call.

## Verification phase
1. Upload 10k-row CSV. Job runs; UI polling shows progress.
2. Final state: 9,950 inserted, 50 skipped (duplicates), errors.csv lists 50 rows with reason.
3. Upload malformed CSV (missing required col). Job fails fast with clear error.
4. Upload 1M-row CSV. Memory stays under 200MB throughout.
5. Custom-data column mapped, validated, stored as JSON on lead.

## Test phase
**Unit:** row normalization (phone, dates, custom fields); error formatting.
**Integration:** end-to-end upload → DB rows + error report.
**Run:** `cd workers && npm test`

## Acceptance criteria
- [ ] 1M-row CSV imports without OOM.
- [ ] ≥ 1,000 rows/sec on dev hardware.
- [ ] Progress visible in UI within 1s of starts.
- [ ] Error report has line number + reason per row.
- [ ] Duplicates within list silently skipped (not error).
- [ ] Header mapping persisted per list; reused on next upload.
- [ ] Job retried up to 3x on transient failure (S3 read error).

## Risks
- **Header inference wrong** → user gets wrong-mapped data. Mitigate: mapping confirmation step in UI before processing starts.
- **Memory leak in streaming parser** — pin a known-good version; load-test.
