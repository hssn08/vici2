# O01 — Observability Research (Prometheus + Grafana + Alertmanager)

Status: RESEARCH (blocked on F01). Do NOT advance to PLAN. No yaml/json artifacts produced here.
Scope: metrics + alerts only. Logs and traces are explicitly out of scope for O01 v1; covered in §8 below.
Date: 2026-05-06.

## 1. Executive summary (10 bullets)

1. **Stack selection.** Prometheus 3.x server (3.0 GA Nov 2024 [1][2]) + Alertmanager 0.27+ [10] + Grafana 11.x [12] running as a separate `docker-compose.observability.yml` overlay so the obs stack can be torn down independently of the dialer plane. All three speak well-known protocols and are CNCF-blessed [1].
2. **Scrape topology = pull, single-tenant Phase 1.** One Prometheus instance scrapes every `/metrics` endpoint listed in F01 (api:9101, dialer:9102, workers:9103, plus FreeSWITCH-exporter:9282, mysqld_exporter:9104, redis_exporter:9121, node_exporter:9100). 15 s default scrape interval; 5 s for the dialer (drop-rate accounting moves fast). No federation, no Thanos, no remote-write in Phase 1 — single Prometheus on the host, 30 day retention.
3. **Native histograms by default for new histograms.** Prometheus 3.0 makes native (sparse exponential) histograms a first-class citizen with out-of-order ingestion, RW2 metadata, and PromQL helpers (`histogram_count`, `histogram_sum`, `histogram_quantile`, `histogram_avg`, `histogram_fraction`) [3][4]. We instrument with `prometheus/client_golang` (Go) and `siimon/prom-client` (Node). For latency-style metrics where a fixed SLO bucket is required (drop-rate, ESL event latency), we use **NHCB** (Native Histograms with Custom Buckets) so PromQL stays consistent and we can later flip to pure native [4][13]. We dual-emit classic histogram for the first release behind `always_scrape_classic_histograms: true` so existing dashboards keep rendering during the migration window.
4. **UTF-8 metric names allowed but we don't use them.** Prometheus 3.0 supports UTF-8 in metric and label names [1][2]. We deliberately stay on the ASCII subset (`vici2_<subsystem>_<unit>`) per SPEC.md §3.6 to keep compatibility with older client libs, alert tooling, and grep-based searches.
5. **Grafana dashboards as JSON-in-git, generated by grafonnet (go-jsonnet).** Each dashboard's source of truth is a `.libsonnet` file under `grafana/src/`, compiled to plain `.json` checked into `grafana/dashboards/` and provisioned via the Grafana file provider. Reasoning in §3 — rejected pure-JSON (drift) and Terraform provider (overkill at single-tenant) [5][6][12]. Recording rules push expensive PromQL into Prometheus so Grafana only renders sums/avgs.
6. **FreeSWITCH metrics: we write a small in-house ESL exporter.** The Rust `mod_prometheus` is community/experimental and not production-ready [7]; the three Go/Python ESL-based community exporters (`florentchauveau`, `mroject`, `znerol`) [7][8] cover only generic counts. We need richer, conference-aware metrics (per-campaign call counts, per-conference participants, ESL event-loop latency). Plan: implement `freeswitch-exporter` in Go reusing the same eslgo-based connection pool the dialer engine ships with (T01), exposing `vici2_freeswitch_*` and `vici2_esl_*` series. ~300 LoC, low risk.
7. **Cardinality discipline is non-negotiable.** Per SPEC §3.6 + Prometheus best practice [9][14], we will NEVER label per-call_uuid, per-agent_id, per-lead_id, per-phone-number on any metric. State counts use `state="paused"|"ready"|...` (bounded set), dispositions use the bounded disposition catalogue, and per-campaign series are admitted only because campaign count is bounded (~tens, not millions). Recording rules pre-aggregate everything dashboards need.
8. **Alerts are tied to compliance hard floors (SPEC §4.1).** Drop-rate has TWO thresholds: 2.5% / 5 m → WARN (Slack); 2.9% / 1 m → PAGE (PagerDuty). Both fire well before the 3.0% FCC ceiling [11]. DNC scrub failures, TCPA outside-window attempts, missing recording-consent rows are PAGE-on-first-occurrence (zero tolerance). FS down, ESL disconnect, MySQL replication lag, hopper starvation are operational PAGE.
9. **Routing tree.** Alertmanager routes on `severity=page → PagerDuty + #ops-pager Slack`, `severity=warn → #ops-warn Slack only`, `severity=info → email digest`. Inhibition rule: `freeswitch_down` PAGE inhibits all derived alerts (drop-rate, hopper, ESL). Silence patterns documented for planned maintenance windows [10].
10. **Tracing deferred.** OpenTelemetry traces are NOT in O01 v1. SPEC.md does not require them; they would double instrumentation surface and we have no current debugging gap that traces solve. Phase 2 candidate: OTLP traces from API + dialer + a SIP-aware collector [15] once we have a recurring debug case. Logs: stdout JSON in Phase 1; Loki + Promtail/Alloy is a Phase 2 module (not O01).

## 2. Prometheus version + scrape topology

### 2.1 Version

Prometheus **3.x** (3.0 GA Nov 2024) [1][2]. Pinned to the latest patch in compose. Notable 3.x features we rely on:

- **Native histograms (stable).** Sparse exponential bucketing; far cheaper than classic histograms with comparable resolution [3].
- **Out-of-order ingestion** (extended to native histograms in 3.0). Critical for the dialer engine where pacing tick events may arrive slightly out of monotonic order under restart/reconcile [3].
- **Remote-Write 2.0** (RW2) protocol with metadata, exemplars, created-timestamp, native histogram support, string interning [1][2]. We keep `remote_write` disabled in Phase 1 but design with RW2 as the future export protocol when we add Thanos/Mimir.
- **UTF-8 metric & label names.** We do NOT use the new charset; SPEC §3.6 mandates ASCII `vici2_<subsystem>_<unit>` [1].
- **New PromQL engine** (experimental) — leave on default engine until 3.x line stabilises it.

Storage: local TSDB; `--storage.tsdb.retention.time=30d` matches the FCC 30-day rolling drop-rate window [11]. WAL on the same disk is acceptable at single-host scale.

### 2.3 Recording rules (skeleton — not authored here)

Computed in Prometheus, exposed to Grafana so dashboards stay snappy [18][19]:

- `vici2:dialer_drop_ratio_30d:rate` — 30-day numerator/denominator over `vici2_dialer_abandoned_calls_total` and `vici2_dialer_live_answered_total`, per `campaign`.
- `vici2:agent_state_count:by_state` — instant gauge sum of `vici2_agent_state_count` by `state`.
- `vici2:freeswitch_calls_active:sum` — over all FS instances (Phase 4 sharding-ready).
- `vici2:esl_event_latency_p99:5m` — `histogram_quantile(0.99, sum by (le) (rate(vici2_esl_event_latency_seconds_bucket[5m])))` (NHCB-aware once classic dropped).

Recording rules go in `prom/rules/recording/*.yml`; alerts in `prom/rules/alerts/*.yml` (file split keeps reviews tight).

## 3. Grafana version + dashboard-as-code

### 3.1 Version

Grafana **11.x OSS** [12]. Phase 1 uses the `grafana/grafana-oss` Docker image. 11.3 added Grafana-managed recording rules (we still prefer Prometheus-side recording rules; see §3.3) [18].

### 3.2 Dashboard-as-code choice: grafonnet

We considered three options:

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| **Pure JSON checked-in** | zero deps, GUI-friendly | drift between dashboards, ~10 KB JSON per panel, no DRY | Reject — 8 dashboards × ~12 panels × shared panels (drop%, alert summary) = copy-paste hell |
| **Terraform Grafana provider** | unified IaC, can manage folders/users/data sources [6] | overkill at single-tenant scale; need Terraform state mgmt; HCL is awkward for nested panel config | Reject for Phase 1; revisit Phase 4 for multi-tenant |
| **grafonnet (jsonnet)** [5] | DRY, panel functions, latest schema generated from grafana-foundation-sdk; outputs plain JSON Grafana provisioner consumes | learning curve; jsonnet syntax | **CHOSEN** |

**Toolchain.** `go-jsonnet` (NOT C jsonnet; Grafana strongly recommends the Go impl for performance [5]). `jsonnet-bundler` (`jb`) for deps. Compile in CI: `make grafana-build` runs `jsonnet -J vendor/ -m grafana/dashboards/ grafana/src/*.jsonnet`. Compiled JSON is committed (so Grafana provisioning works without a build step at runtime).

Provisioning: `grafana/datasources/prometheus.yml` declares Prometheus as default DS; `grafana/provisioning/dashboards/file.yml` points at `grafana/dashboards/`.

### 3.3 Performance posture [18][19]

- Hard limit ≤ 20 panels per dashboard. Exec summary stays under 12.
- Use Prometheus recording rules for any PromQL that would otherwise iterate raw histogram buckets (drop ratio, p99 latencies). Grafana renders pre-computed series.
- Use template variables backed by `label_values()` — never embed full PromQL in variable defs.
- Time range default = 1 h on operational dashboards, 6 h on exec; users can extend.
- Collapsible rows for less-hot sections so panels don't render until expanded.

## 4. Per-service exporter strategy

### 4.1 Go services (dialer)

`prometheus/client_golang` [4]. Standard pattern: `promhttp.Handler()` mounted at `/metrics` on a separate port (9102) so internal metrics aren't exposed via the dialer's gRPC port. Default Go runtime collectors (goroutines, GC, mem) are emitted automatically.

Histograms in dialer use **NativeHistogramBucketFactor=1.1** (≈10% relative resolution) per recommendation in [4]; classic histograms dual-emitted only where the SLO requires fixed buckets (drop-rate window: 0, 1.0s, 1.5s, 2.0s, 2.5s — the 2-second cutoff is the FCC abandonment definition [11]).

### 4.2 Node services (api, workers, web optional)

`siimon/prom-client` [21] wrapped by `fastify-metrics` [16] for the api. Default route metrics: HTTP duration histogram, request count, response size. We override default buckets to match our latency profile (api targets sub-200 ms): `[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5]`. `workers` (no Fastify) uses `prom-client` directly with `register.metrics()` served via a tiny `http.createServer` on 9103.

### 4.3 FreeSWITCH — in-house ESL exporter

Decision: **write `freeswitch-exporter` ourselves**.

Rationale:
- `friends-of-freeswitch/mod_prometheus` (Rust) is explicitly NOT production-ready [7] and forces a Rust toolchain into the FS image.
- Existing Go/Python ESL community exporters [7][8] expose generic numbers (calls active, sessions/sec) but don't know about our conference-per-agent primitive (DESIGN §1.3) — we'd still need custom code for `vici2_agent_conference_participants` etc.
- We're already shipping `eslgo` in the dialer engine (T01). Reusing the same lib in a sibling binary is cheap and means one bug class to maintain.

Sketch (no code yet — RESEARCH only):
- Inbound ESL connection (port 8021) + event subscription for `CHANNEL_CREATE/DESTROY/ANSWER`, `CONFERENCE_*`, `CUSTOM mod_callcenter`.
- Periodic poll (1 s) of `api show calls count`, `api show channels count`, `api conference list`, `api status` for gauges.
- Histogram of ESL event ingest latency (`now() - Event-Date-Timestamp`) — feeds the ESL latency alert.
- Exposes `vici2_freeswitch_*` and `vici2_esl_*` on `:9282/metrics`.
- Reconnect with exponential backoff; emit `vici2_freeswitch_exporter_recovered_total` per SPEC §4.7.

### 4.4 MySQL — `prometheus/mysqld_exporter` [17][22]

Enabled collectors (modular, per [17][22]):
- `--collect.global_status`, `--collect.global_variables`, `--collect.info_schema.innodb_metrics`, `--collect.info_schema.innodb_cmp`, `--collect.info_schema.innodb_cmpmem`
- `--collect.perf_schema.eventsstatementssum` (lightweight summary; NOT the per-digest collector which can blow cardinality on a busy DB)
- `--collect.slave_status` (Phase 4 replica)
- `--collect.engine_innodb_status` (gated; expensive on busy server)

Disable: `--no-collect.info_schema.processlist` (high cardinality on user/host), `--no-collect.perf_schema.eventswaits`.

Mostly default dashboards from grafana.com/dashboards #7362 / #14057 work; we customise to add `vici2_db_*` panels.

### 4.5 Redis — `oliver006/redis_exporter`

Defaults are fine for KeyDB/Valkey (latter is binary-compatible). We add `--check-keys=t:*:agent:*:state,t:*:hopper:*` to surface per-tenant key counts as gauges (cardinality bounded by tenant × agent count, which is small in Phase 1). Standard dashboard #763 imported as a baseline, then forked to grafonnet under our naming.

### 4.6 Hosts — `node_exporter`

Stock collectors; disable `textfile`, `wifi`, `ipvs` (irrelevant). Standard dashboard #1860 baseline.

### 4.7 Postgres-style (deferred)

Not in scope. If we ever migrate any subsystem to Postgres, `prometheus-community/postgres_exporter` is the equivalent. Noted for future.

## 6. Alert rules tied to compliance gates

All thresholds set BELOW the regulatory ceiling so we alert before breach. PAGE = pages oncall; WARN = chat only; INFO = digest email.

### 6.1 Compliance hard-floor alerts (SPEC §4.1)

| Alert | Expr (informal) | For | Severity | Rationale |
|---|---|---|---|---|
| `Vici2DropRateApproachingFCC`  | `vici2:dialer_drop_ratio_30d:rate > 0.025` | 5 m | **warn** | FCC ceiling 3% over 30 d/campaign [11]; alert at 2.5% gives runway |
| `Vici2DropRateImminentBreach`  | `vici2:dialer_drop_ratio_30d:rate > 0.029` | 1 m | **page** | 0.1% headroom before regulatory breach |
| `Vici2DNCScrubFailure`         | `increase(vici2_dialer_dnc_scrub_failures_total[5m]) > 0` | 0 m | **page** | Any failure is a compliance event; statutory damages $500–1500/call [11] |
| `Vici2TCPAOutsideWindowAttempt`| `increase(vici2_dialer_tcpa_outside_window_attempts_total[5m]) > 0` | 0 m | **page** | 8 am–9 pm gate per SPEC §4.1 |
| `Vici2RecordingConsentMissing` | `increase(vici2_compliance_recording_consent_state_missing_total[5m]) > 0` | 0 m | **page** | 2-party-consent jurisdictions |

### 6.2 Operational alerts

| Alert | Expr | For | Severity |
|---|---|---|---|
| `Vici2FreeSwitchDown`        | `vici2_esl_connection_up == 0` | 30 s | page |
| `Vici2ESLDisconnect`         | `increase(vici2_esl_reconnects_total[5m]) > 3` | 5 m | warn |
| `Vici2ESLEventLatencyHigh`   | `histogram_quantile(0.99, ... vici2_esl_event_latency_seconds_bucket ...) > 0.5` | 5 m | warn |
| `Vici2HopperStarved`         | `vici2_dialer_hopper_depth < (vici2_agent_state_count{state="ready"} * 1.5)` | 2 m | warn |
| `Vici2HopperFloor`           | `vici2_dialer_hopper_depth == 0 and vici2_agent_state_count{state="ready"} > 0` | 1 m | page |
| `Vici2MySQLDown`             | `mysql_up == 0` | 1 m | page |
| `Vici2MySQLReplicationLag`   | `mysql_slave_lag_seconds > 30` | 5 m | warn |
| `Vici2RedisDown`             | `redis_up == 0` | 30 s | page |
| `Vici2DialerErrorRateHigh`   | `rate(vici2_dialer_errors_total[5m]) > 1` | 5 m | warn |
| `Vici2APIErrorRate5xx`       | `sum(rate(vici2_http_requests_total{status=~"5.."}[5m])) / sum(rate(vici2_http_requests_total[5m])) > 0.01` | 5 m | warn |

### 6.3 Routing tree (informal — to be expressed in PLAN)

```
root
├── match: severity=page → receiver: pagerduty + slack:#ops-pager
│   └── group_by: [tenant_id, alertname, campaign]
│       group_wait: 10 s, group_interval: 5 m, repeat_interval: 4 h
├── match: severity=warn → receiver: slack:#ops-warn
│   └── group_wait: 30 s, repeat_interval: 12 h
└── match: severity=info → receiver: email-digest
    └── repeat_interval: 24 h
```

Inhibition rules (per [10]):
- `Vici2FreeSwitchDown` (page) inhibits `Vici2ESLDisconnect`, `Vici2ESLEventLatencyHigh`, `Vici2HopperStarved`, `Vici2HopperFloor`, drop-rate alerts (since with FS down, denominators are 0).
- `Vici2RedisDown` inhibits `Vici2HopperStarved` / `Vici2HopperFloor` (hopper lives in Redis).
- `Vici2MySQLDown` inhibits `Vici2DropRateApproachingFCC` / `Vici2DropRateImminentBreach` because numerators are written through MySQL.

Silences: planned-maintenance pattern documented in runbook with `matchers` like `tenant_id=1, env=staging`. Encourage time-bounded silences (max 2 h) via Alertmanager UI to avoid forever-silenced alerts [10].

## 7. Cardinality discipline

Per [9][14] and SPEC §3.6:

**Forbidden labels** (would explode cardinality, no exceptions):
- `call_uuid`, `session_uuid`, `b_leg_uuid`
- `agent_id`, `user_id`, `lead_id`
- `phone_number`, `caller_id`, `dnis`
- `request_id`, `trace_id` (these belong on logs / traces, not metrics)
- `email`, `ip_address`

**Allowed labels** (bounded sets, audited):
- `campaign` (~tens)
- `tenant_id` (single-tenant Phase 1; future-bounded)
- `state` (enum: `ready|paused|incall|wrapup|loggedin|loggedout`)
- `disposition` (catalog table; bounded ~50)
- `pause_code` (catalog table; bounded)
- `carrier` (single-digit)
- `result`, `outcome`, `reason` (small enums)
- `route`, `method`, `status` (HTTP — bounded by route table size, not free-form path)
- `code` (error-code catalog from SPEC §3.5)
- `kind` (small enums like `agent|customer`)

**Cardinality budget.** At Phase 1 scale (1 tenant, 10 campaigns, ~100 agents, 6 services × ~30 metrics each), expected total active series < 50 k — well inside Prometheus single-host comfort zone (rule of thumb: stay under 10M [14], we're 200× under that). At Phase 4 (multi-tenant, 1k agents, sharded FS) we re-evaluate; if approaching 1M series, revisit federation/Mimir.

**Enforcement.** A `cardinality_check` CI step runs `promtool test rules` AND queries `count(count by (__name__)({__name__=~"vici2_.+"}))` against an in-CI scrape of stub services; fails if any single metric exceeds 10 k series. Bad regressions caught at PR time.

## 8. Tracing / logging scope

### 8.1 Metrics-only in O01 v1 (DEFER tracing)

Recommendation: do NOT add OpenTelemetry tracing in Phase 1.

Reasons:
- O01.md Goal explicitly says "metrics" only. Adding traces doubles instrumentation surface for marginal Phase 1 value.
- The metrics + structured logs combination already covers our top debug cases (lost call, dropped event, slow dialer tick).
- VoIP/SIP tracing through OTel exists [15] but requires either a Kamailio/OpenSIPS-style middleware (we don't run one) or custom hooks in `mod_sofia` — neither cheap.
- Phase 2 will likely surface a recurring debug case (e.g., "why did this 3-way fail across two FS shards") where traces become worth the cost. Add then, not now.

### 8.2 Logs out-of-scope for O01

Logs are stdout JSON (SPEC §3.4) collected by Docker's logging driver in dev. Centralized log aggregation (Loki + Promtail or Grafana Alloy) is a separate Phase 2 module — not O01.

### 8.3 If/when we add tracing (Phase 2 sketch)

- OTel SDK in Go (dialer) and Node (api). Auto-instrumentation for HTTP, gRPC, MySQL, Redis.
- Custom spans for: `originate`, `bridge`, `dispose`, `dnc-scrub`, `tz-gate`.
- Carry `tenant_id` on every span as an attribute (mirrors §4.5 in SPEC).
- OTLP collector → Tempo (Grafana stack) or Jaeger (independent). Both viable.
- Sampling: head-based 10% baseline + tail-based 100% for any trace with an error span.

## 10. Open questions for PLAN

1. **Drop-rate denominator window.** FCC says "30-day rolling per campaign" [11]. Implementation: do we compute via `increase()` over 30 d (Prometheus retention must match) or via a MySQL-backed materialised counter (E05's territory) that we then expose via a tiny exporter? PLAN must pick one.
2. **Native histogram readiness across stack.** Are all our datasources (Grafana 11, Alertmanager 0.27, recording-rule machinery) ready for pure-native histograms, or do we keep classic dual-emit indefinitely? Answer affects bucket counts shipped.
3. **NHCB vs pure native for compliance histograms.** For drop-rate buckets we need an exact 2.0 s threshold (FCC definition). NHCB is the current consensus [4][13]; confirm.
4. **FreeSWITCH-exporter reconcile semantics.** On ESL reconnect, should the exporter zero gauges (risking dashboards flickering) or hold last-known values until first successful poll? PLAN call.
5. **Multi-FS gauge aggregation.** When we shard FS in Phase 4, sums-of-gauges across instances need consistent labeling (`instance=fs01.example.com` vs `shard=01`). Define labelling convention now.
6. **Recording rule placement.** Prometheus-side vs Grafana-managed (11.3+) [18][19]. Default Prometheus-side for portability; revisit if Grafana-managed gives material UX wins.
7. **Alert routing to PagerDuty escalation policy.** Single team or split (compliance vs ops)? Need oncall org context to finalise.
8. **Maintenance window primitive.** Encode as Alertmanager silence (operator action) or as a Prometheus `inhibit` matcher driven by a `vici2_maintenance_active{component=...}` gauge written by a deploy script? Latter is more auditable.
9. **Loki/Phase 2 logging — separate module ID.** Should be filed as O02-or-similar, NOT folded into O01. Confirm in PLAN review.
10. **Tracing decision gate.** Define explicit criteria that would unblock Phase-2 OTel adoption (e.g., "≥ 3 incidents/quarter where root-cause-analysis took > 1 hour due to lack of distributed trace").

## 12. Runbook structure (preview for PLAN)

For each PAGE alert, the runbook entry under `spec/runbooks/oncall.md` will follow this template:

```
### <AlertName>
- TRIGGERED BY: <metric expr summary>
- WHY IT MATTERS: <link to compliance section / SLO>
- FIRST CHECK: <Grafana dashboard URL + panel>
- LIKELY CAUSES: <ranked list>
- MITIGATION: <ordered steps; first step must be safe (silence/throttle), not destructive>
- ESCALATION: <who to wake at T+15m>
- POST-MORTEM REQUIRED IF: <severity threshold>
```

Order matters: every PAGE runbook must lead with a "first check" Grafana panel link so oncall has signal in <30 s.
