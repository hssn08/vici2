## 1. Executive summary (10 bullets)

1. **IVR engine = DTMF-driven decision tree routing inbound calls.** An IVR is a directed graph: every node plays a prompt + collects DTMF digits; edges map collected input to the next node or a terminal action. Terminal actions are: route to an I01 in-group (`transfer ingroup_{id} XML default`), hangup with apology, route to voicemail (I05), external transfer (PSTN), or offer a callback (D06). Phase 1 caps tree depth at 3 levels; deeper trees are Phase 2.

2. **FreeSWITCH `read` app is the DTMF-collection primitive.** `read <min> <max> <sound-file> <variable-name> <timeout-ms> <terminators>` blocks the dialplan until digits are collected or timeout fires. The result is stored in a channel variable which the next dialplan extension branches on. This is the simplest correct approach — no external controller needed for static trees (RESEARCH §3).

3. **Phase 1 IVR: static FreeSWITCH XML dialplan generated by the API.** For static trees (operator configures via admin UI → API renders XML → `bgapi reloadxml`), the FreeSWITCH dialplan XML is the right execution engine. It runs entirely inside FS; no round-trips needed during the call. The API service owns an `IvrRenderer` that reads `ivrs`, `ivr_nodes`, `ivr_edges` from MySQL and writes `freeswitch/conf/dialplan/public/70_ivr_{id}.xml` (in the slot reserved by F03 HANDOFF §5.2). RESEARCH §2 compares three approaches and selects this.

4. **Phase 2 IVR: Go ESL controller for dynamic/conditional trees.** When menus need per-call dynamic data (e.g., "your account balance is $X, press 1 to pay") or complex conditional branching beyond static digit matching, the right architecture switches to a Go ESL controller (`dialer/cmd/ivrbridge/`) that executes node logic over ESL, issuing `bgapi uuid_broadcast` + `play_and_get_digits` and receiving DTMF events. This is the mod_lua / mod_esl philosophy but in Go, matching the project's existing `eslbridge` binary. RESEARCH §2.3 covers the transition path.

5. **Prompt storage: S3-backed WAV/MP3 files with a local FS cache.** Uploaded prompts are stored in S3 under `ivr/{tenant_id}/{ivr_id}/{prompt_id}.{ext}`. FreeSWITCH plays them via the `playback` application; the dialplan generator writes the full path. A startup script syncs S3 → `/var/lib/freeswitch/sounds/ivr/` (read-only bind-mount in dev; S3-fuse or rclone sync in prod). RESEARCH §4.

6. **DID assignment adds `ivr_id` to `did_numbers`.** A DID's `route_kind=ivr` and `route_target` already exist in the schema (`DidRouteKind.ivr`). I02 adds the IVR lookup: the public dialplan entry point reads `route_target` as the IVR id, then transfers to `ivr_{id}_entry` in the default context where the XML tree begins. RESEARCH §6.

7. **Multi-language: per-node language variants.** Each `ivr_prompt` row has a `lang` column (`en`, `es`, `fr`). The IVR node plays `prompts WHERE lang = chosen_lang`. Language is determined at call entry: either from the DID config (`did_numbers.default_lang`) or from an initial language-selection menu (a special node type: `lang_select`). Selected language is stored in channel var `vici2_ivr_lang` and consulted by every subsequent node. RESEARCH §6.

8. **Analytics: `ivr_traversal_log` table, monthly-partitioned.** One row per node visit: `(ivr_id, session_uuid, node_id, lang, digit_entered, duration_ms, timestamp)`. This enables per-node drop-off funnel analysis. The Go IVR service (Phase 2 controller) writes these rows; in Phase 1 (static XML), the dialplan emits `bgapi uuid_setvar` to write a summary JSON into a channel variable and the HANGUP event triggers a Node.js Fastify webhook that inserts the traversal log rows. RESEARCH §8.

9. **DNC: inbound calls are exempt from the federal/state DNC hot path** (TCPA/TSR DNC applies to outbound solicitations; inbound calls are caller-initiated). However, I02 still logs the inbound call against the caller's phone number in a lightweight audit note (no Bloom query, no MySQL DNC check). This matches D05 PLAN §0 bullet 10 ("inbound: exempt but still log"). RESEARCH §5.

10. **Failure modes: missing prompt plays silence + falls through.** If a prompt file is missing, the `read` app skips playback and still waits for DTMF for the configured timeout. The IVR renderer validates all prompt paths at render time and refuses to write invalid XML; this is the primary defense. Secondary: a watchdog extension fires `hangup NORMAL_CLEARING` after `invalid_max` consecutive invalid inputs. RESEARCH §9.

## 2. IVR engine approach comparison

### 2.1 Option A — FreeSWITCH dialplan XML (selected for Phase 1)

FreeSWITCH's dialplan XML can implement a complete IVR natively using `<action application="read">` + `<condition field="${variable}">` branching. Extensions per node, with `<action application="execute_extension">` for subtree traversal.

**Strengths:**
- Zero external round-trips during the call. All state is in FS channel variables.
- Extremely low latency (DTMF collection and branching happen inside FS with no network hops).
- Proven: every Asterisk-based IVR (including Vicidial's) uses this pattern.
- Existing F03 infrastructure: `bgapi reloadxml` is already the reload mechanism for all dialplan changes (F03 HANDOFF §2, I01 PLAN §15).
- Simple failure model: if the API is down, existing loaded XML continues serving calls.

**Weaknesses:**
- XML generation must be done offline (before the call). Cannot compute per-call dynamic data.
- Deep nesting creates verbose XML (100-node tree = ~3000 lines of XML). Manageable for Phase 1 (≤3 levels ≈ ≤15 nodes per tree).
- Language branching requires XML duplication per language variant.

**Verdict:** Correct for Phase 1 (static trees, ≤3 levels). Generator lives at `api/src/services/ivr/IvrRenderer.ts`.

### 2.2 Option B — mod_lua (rejected for Phase 1)

mod_lua provides a scripting layer inside FS where Lua scripts can call `session:read()`, branch, and issue `session:transfer()`. Used in many production IVR deployments.

**Why rejected for Phase 1:**
- Requires shipping Lua scripts into the FS container at deploy time — complicates CI/CD.
- Go team has no Lua expertise; introduces a second language for FS logic.
- Lua is single-threaded per session (fine) but debugging Lua inside FS is harder than debugging Go.
- For Phase 2, Go ESL controller is preferred over Lua because (a) it shares the `eslgo` library already pinned in T01 PLAN §1; (b) it can be unit-tested without a running FS instance; (c) it can integrate with the existing Valkey/MySQL infrastructure directly.

### 2.3 Option C — Go ESL controller via `eslbridge` (selected for Phase 2)

For Phase 2, a new `dialer/cmd/ivrbridge/` binary connects to FS via ESL (`eslgo` v1.5.0, T01's pinned library). On an incoming call (CHANNEL_ANSWER event), ivrbridge takes control:
1. Issues `bgapi uuid_setvar {uuid} vici2_ivr_id {ivr_id}` to tag the channel.
2. Executes node logic: `bgapi uuid_broadcast {uuid} execute::play_and_get_digits '{args}'`.
3. Collects DTMF via DTMF event subscription (ESL `RegisterEventListener("DTMF", handler)`).
4. Evaluates edge conditions against collected digit.
5. Recurses to next node or executes terminal action (`bgapi uuid_transfer {uuid} ingroup_{igid} XML default`).

**Transition from Phase 1 → Phase 2:**
- Phase 1 XML dialplan files `70_ivr_{id}.xml` are still in place as fallback.
- Phase 2 ivrbridge subscribes to CHANNEL_ANSWER events for calls that have `vici2_ivr_id` set in the public context entry point.
- Phase 1 XML can be disabled per-IVR by setting `ivrbridge_controlled=true` on the IVR row.
- No flag day: Phase 1 and Phase 2 can coexist (simple IVRs stay in XML; dynamic IVRs use ivrbridge).

### 2.4 mod_xml_curl approach (rejected)

F03 HANDOFF §9 ships `xml_curl.conf.xml` with EMPTY `<bindings/>`. Using `mod_xml_curl` for dialplan lookup (FS calls the API on every call to get the dialplan XML) is appropriate for very dynamic environments but adds per-call HTTP latency (~10 ms) and a dependency on API availability for every inbound call. For IVR specifically, the static XML approach is strictly superior: the XML is pre-rendered and the call proceeds even if the API is down. mod_xml_curl is reserved for I02's Phase 2+ or for cases where the IVR tree must change faster than a reloadxml cycle (sub-second changes). DEFERRED.

## 3. FreeSWITCH `read` app — DTMF collection

The FS `read` application signature:
```
<action application="read" data="<min_digits> <max_digits> <sound_file> <var_name> <timeout_ms> <terminators>"/>
```

- `min_digits`: minimum number of digits to collect before returning (usually 1 for menu selection).
- `max_digits`: maximum digits (1 for single-digit menu; 10 for phone number entry).
- `sound_file`: the prompt to play while waiting. If blank string `''`, silence is played (prompt played before `read` via `playback`).
- `var_name`: channel variable to store the result. Convention: `ivr_digit_{node_id}`.
- `timeout_ms`: time to wait after last digit (or after prompt ends if min=1). Default recommendation: 5000ms for menu selection; 3000ms after last digit for multi-digit.
- `terminators`: `#` to end early (standard phone convention); `none` for single-digit menus.

**Timeout handling:** If `read` returns an empty variable (timeout or no input), the generated extension checks `${ivr_digit_{node_id}} == ''` and routes to the timeout edge. If a node has been reached `invalid_max` times (tracked via `${ivr_invalid_count}` channel var), it routes to the hangup terminal.

**Example generated XML for a 2-digit menu node:**
```xml
<!-- Node: main_menu (ivr_id=5, node_id=12) -->
<extension name="ivr_5_node_12" continue="false">
  <condition field="destination_number" expression="^ivr_5_n12$">
    <action application="set" data="ivr_invalid_count=0"/>
    <action application="playback" data="/var/lib/freeswitch/sounds/ivr/t1/5/12_en.wav"/>
    <action application="read" data="1 1 '' ivr_digit_12 5000 none"/>
    <action application="execute_extension" data="ivr_5_n12_branch XML default"/>
  </condition>
</extension>

<!-- Branch dispatcher for node 12 -->
<extension name="ivr_5_node_12_branch" continue="false">
  <condition field="${ivr_digit_12}" expression="^1$">
    <action application="transfer" data="ivr_5_n20 XML default"/>
  </condition>
  <condition field="${ivr_digit_12}" expression="^2$">
    <action application="transfer" data="ivr_5_n21 XML default"/>
  </condition>
  <condition field="${ivr_digit_12}" expression="^$">
    <!-- timeout -->
    <action application="transfer" data="ivr_5_n12_timeout XML default"/>
  </condition>
  <condition field="${ivr_digit_12}" expression="^.*$">
    <!-- invalid -->
    <action application="set" data="ivr_invalid_count=${expr(${ivr_invalid_count}+1)}"/>
    <action application="execute_extension" data="ivr_5_n12_invalid_check XML default"/>
  </condition>
</extension>
```

## 4. Prompt storage and delivery

### 4.1 Storage path convention

S3 bucket: `vici2-media` (same bucket as recordings but under different prefix).

S3 prefix: `ivr/{tenant_id}/{ivr_id}/{prompt_id}_{lang}.{ext}`

Example: `ivr/1/5/12_en.wav` → English prompt for node 12 of IVR 5, tenant 1.

FS local path: `/var/lib/freeswitch/sounds/ivr/t{tenant_id}/{ivr_id}/{prompt_id}_{lang}.wav`

### 4.2 Format requirements

- Audio: 8kHz or 16kHz mono WAV (PCM 16-bit) for broadest FS codec compatibility.
- MP3 accepted at upload; converted to WAV by the API upload pipeline via `ffmpeg -ar 8000 -ac 1 -acodec pcm_s16le`.
- Maximum file size: 10 MB per prompt (enforced at upload).
- Maximum prompt duration: 120 seconds (enforced post-conversion).

### 4.3 Sync to FS containers

Development: `freeswitch/sounds/ivr/` directory bind-mounted into the FS container (docker-compose.dev.yml volume mount, same pattern as F03 HANDOFF §6 for recordings).

Production (Phase 2): rclone sync job runs every 60s (`ivr_sync` sidecar), or S3-fuse mount (read-only). The IVR renderer validates prompt existence in S3 at render time; if a prompt is missing it rejects the save and returns a validation error to the admin UI.

### 4.4 Default system prompts

The API ships a default set of system prompt WAV files for common IVR messages:
- `sys_invalid.wav` — "I'm sorry, that was an invalid entry."
- `sys_timeout.wav` — "I'm sorry, I didn't receive your input."
- `sys_goodbye.wav` — "Thank you for calling. Goodbye."
- `sys_callback_offer.wav` — "Press 1 to receive a callback. Press any other key to continue waiting."

These live at `/var/lib/freeswitch/sounds/ivr/sys/` and are not tenant-specific.

## 5. DNC and inbound calls

Per TCPA 47 CFR 310.4(b)(1)(iii) and TSR, the National DNC Registry applies to outbound *telephone solicitations*. An inbound call initiated by the consumer is not a telephone solicitation and is not subject to the DNC registry. Reference: FTC TSR FAQ §37 ("The Rule does not restrict sellers or telemarketers from receiving inbound calls from consumers").

**I02 behavior:**
- Does NOT invoke D05 `CheckDNC` on the inbound DNIS or caller-ID.
- Does NOT consult the Valkey Bloom filters (`bf:dnc:federal`, `t:{tid}:dnc:internal:bloom`).
- DOES write a `call_log` row (managed by R01) with `direction=in`, which indirectly preserves a record of the inbound call for audit purposes.
- The caller's phone number IS checked against internal DNC only if the inbound call triggers an outbound callback (D06 scheduling); at that point D05 runs its normal 5-gate check on the outbound leg.

**Edge case: caller says "put me on your do-not-call list."** Agent dispositions the call; D04 triggers the internal DNC add (same path as outbound DNC disposition). I02 is not involved.

## 6. Multi-language IVR

### 6.1 Language detection modes

**Mode A: DID-level default.** `did_numbers.default_lang` (VARCHAR 5, e.g., `'en'`, `'es'`) sets the language for all calls to that DID. Simple; no caller interaction. Phase 1 default.

**Mode B: Initial language-selection node.** A special node type `lang_select` plays a bilingual greeting ("Para Español, oprima 2. For English, press 1.") and routes digit 1 → set `vici2_ivr_lang=en`, digit 2 → set `vici2_ivr_lang=es`, then continues to the entry node of the same IVR. This is the standard NANP bilingual IVR pattern.

**Mode C: Caller-preferred language from lead record.** On CHANNEL_ANSWER, I02's entry point checks `${vici2_tenant_id}` and performs a `D01.lookupLeadByPhone` (same pattern as I01 queue entry). If the lead has a `preferred_lang` field, it is written into `vici2_ivr_lang`. Phase 2 enhancement.

### 6.2 Channel variable contract

```
vici2_ivr_lang     channel var — lang code chosen for this session; default 'en'
vici2_ivr_id       channel var — IVR tree ID (set at DID entry)
vici2_ivr_node_id  channel var — last node visited (for analytics on HANGUP)
vici2_ivr_path     channel var — colon-delimited list of visited node IDs (analytics)
```

These vars are emitted to the HANGUP event and consumed by the traversal log writer.

### 6.3 Prompt resolution order

For a given node and language:
1. `ivr_prompts WHERE node_id=N AND lang=vici2_ivr_lang` — exact match.
2. `ivr_prompts WHERE node_id=N AND lang='en'` — English fallback.
3. System prompt `sys_invalid.wav` — last resort (plays something, doesn't crash).

Content pattern:

</include>

## 8. Analytics — traversal log

### 8.1 Data model

```sql
CREATE TABLE ivr_traversal_log (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id    BIGINT NOT NULL DEFAULT 1,
  ivr_id       BIGINT NOT NULL,
  session_uuid VARCHAR(40) NOT NULL,
  node_id      BIGINT NOT NULL,
  lang         VARCHAR(5) NOT NULL DEFAULT 'en',
  digit        VARCHAR(8) DEFAULT NULL,   -- NULL on timeout; '' on hangup during prompt
  outcome      ENUM('digit','timeout','hangup','invalid','terminal') NOT NULL,
  duration_ms  INT UNSIGNED NOT NULL,
  entered_at   DATETIME(6) NOT NULL,
  INDEX idx_itl_t_ivr   (tenant_id, ivr_id, entered_at),
  INDEX idx_itl_t_sess  (tenant_id, session_uuid),
  INDEX idx_itl_t_node  (tenant_id, node_id, outcome, entered_at)
) PARTITION BY RANGE (TO_DAYS(entered_at)) (
  PARTITION p_2026_05 VALUES LESS THAN (TO_DAYS('2026-06-01')),
  PARTITION p_2026_06 VALUES LESS THAN (TO_DAYS('2026-07-01')),
  PARTITION p_2026_07 VALUES LESS THAN (TO_DAYS('2026-08-01')),
  PARTITION p_max     VALUES LESS THAN MAXVALUE
);
```

### 8.2 Phase 1 write path (static XML)

In Phase 1, the FS dialplan sets channel vars at each node:
- `vici2_ivr_node_id` = current node id (updated each node).
- `vici2_ivr_path` = colon-delimited list of visited nodes appended at each node (e.g., `"12:20:31"`).
- `vici2_ivr_digits` = colon-delimited list of entered digits matching node order.

On HANGUP_COMPLETE, the ESL bridge (`eslbridge`) reads these channel vars and emits a `CUSTOM vici2.ivr.session_end` event. A Node.js webhook handler (or the `eslbridge` itself) parses the path and digits, then batch-INSERTs rows into `ivr_traversal_log`.

**Limitation:** This approach writes one INSERT per session_end event (reconstructed from path string), not per-node-in-real-time. Acceptable for Phase 1 analytics. Phase 2 ivrbridge writes per-node in real time.

### 8.3 Analytics queries

Key metrics derivable from `ivr_traversal_log`:
- **Node entry count**: `COUNT(*) GROUP BY node_id` — shows which nodes are reached.
- **Drop-off rate per node**: `COUNT(outcome='hangup') / COUNT(*) GROUP BY node_id`.
- **Digit distribution per node**: `COUNT(*) GROUP BY node_id, digit` — shows most-pressed keys.
- **Avg duration per node**: `AVG(duration_ms) GROUP BY node_id` — identifies long-wait nodes.
- **Session completion rate**: sessions where a terminal action was reached vs. hangup in tree.

## 9. Failure modes

### 9.1 Prompt file missing

**Detection:** IvrRenderer validates all prompt S3 URIs before writing XML. If any prompt is missing, the save is rejected with a 422 error listing the missing files.

**Runtime defense (belt + suspenders):** Even if a file disappears after render, FS `playback` on a missing file logs a warning and continues; `read` will still wait for DTMF. The caller hears silence instead of the prompt but can still press digits. If this is detected (O01 alert on FS log `FILE_NOT_FOUND` count), admin is paged.

### 9.2 Invalid input loop

Each node has `invalid_max` (default 3). Channel var `ivr_invalid_count` is incremented on each non-matching digit. When `ivr_invalid_count >= invalid_max`, the extension transfers to the node's `on_invalid_max` target (which can be: hangup, route to a default in-group, or replay the prompt from the start).

```xml
<extension name="ivr_{id}_n{node_id}_invalid_check">
  <condition field="${ivr_invalid_count}" expression="^[3-9]$|^[1-9][0-9]+$">
    <!-- max exceeded: go to terminal -->
    <action application="transfer" data="{on_invalid_max_target} XML default"/>
  </condition>
  <condition field="${ivr_invalid_count}" expression="^.*$">
    <!-- retry: replay node -->
    <action application="transfer" data="ivr_{id}_n{node_id} XML default"/>
  </condition>
</extension>
```

### 9.3 IVR runaway (caller stuck in tree)

`did_numbers.ivr_timeout_sec` (default 300s = 5 minutes) is enforced by a FS `sched_transfer` placed at the DID entry point:

```xml
<action application="sched_transfer" data="+{ivr_timeout_sec} hangup_ivr_timeout XML default"/>
```

The `hangup_ivr_timeout` extension plays `sys_goodbye.wav` and hangs up. This prevents a stuck DTMF collection from holding a channel forever.

### 9.4 API unavailable during IvrRenderer trigger

The IvrRenderer is called synchronously on IVR save (admin UI action). If the API cannot write the XML file (FS container unreachable, file system full), the save is rejected with a 503 and the old XML remains in place. This is a "write-fail-safe" pattern: old config stays live, new config not applied until the error is resolved.

### 9.5 reloadxml during active calls

`bgapi reloadxml` in FS is non-disruptive to calls in flight. Active channel state (channel vars, current extension execution pointer) is not reloaded. Only calls that enter a NEW extension after reload see the new dialplan. This means an IVR update takes effect for new callers within 1–2 seconds (FS reload latency) but does not interrupt callers mid-tree. This is safe by design.

### 9.6 FS pod restart with calls in IVR tree

If FS restarts, active IVR calls are dropped (CHANNEL_HANGUP on all active channels). The `ivr_traversal_log` writer handles this gracefully: it writes the partial path reconstructed from channel vars at the time of hangup. The analytics show `outcome=hangup` mid-tree. No zombie state. The call is gone; caller must redial.

## 10. Vicidial IVR reference

### 10.1 Vicidial's approach

Vicidial implements IVR via Asterisk AGI scripts (`agi/agi-VDAD_ALL_inbound.agi`) and the `extensions.conf` dialplan. The inbound AGI:
1. Answers the call.
2. Looks up the DID in `vicidial_inbound_groups` (MySQL) to find the in-group or IVR group.
3. If the DID is configured for an IVR group (`ivr_group`), calls the IVR handler.
4. The IVR handler plays prompts via AGI `STREAM FILE`, collects DTMF via AGI `GET DATA`.
5. Based on the collected digit, looks up `vicidial_ivr` table for the corresponding action.
6. Recurses for multi-level menus.

### 10.2 Vicidial IVR schema (for reference)

Vicidial's key IVR tables:
- `vicidial_ivr` — parent record: `ivr_id`, `ivr_name`, `active`, `dialplan_number`.
- `vicidial_ivr_options` — one row per menu option: `ivr_id`, `option_value` (the digit), `action` (MENU, INGROUP, DID, AGI, HANGUP), `target` (in-group name or sub-IVR id), `prompt_filename`.

This is essentially a flat table representing a tree. Vici2's design improves on this with explicit node/edge tables that support arbitrary depth, per-node timeouts, invalid-count tracking, and analytics without requiring a separate table join per level.

### 10.3 Key differences from Vicidial

| Aspect | Vicidial | Vici2 I02 |
|---|---|---|
| Storage | MySQL `vicidial_ivr` + `vicidial_ivr_options` (flat) | `ivrs`, `ivr_nodes`, `ivr_edges` (graph model) |
| Execution | Asterisk AGI (Perl) | FS dialplan XML (Phase 1) / Go ESL (Phase 2) |
| DTMF | AGI `GET DATA` | FS `read` app |
| Analytics | None (call_log only) | `ivr_traversal_log` per node |
| Multi-language | Manual (separate IVR per language) | Per-node language variants |
| Admin UI | PHP admin screen, table-based | Web form tree editor (Phase 1), drag-drop (Phase 2) |
| Prompt hosting | Local filesystem on Asterisk server | S3 + local cache |
| Invalid handling | Basic (replay or hangup) | Per-node configurable `invalid_max` + action |

## 11. Dialplan generator architecture

### 11.1 IvrRenderer service

`api/src/services/ivr/IvrRenderer.ts` — TypeScript service that:
1. Reads the IVR tree from MySQL (`ivrs`, `ivr_nodes`, `ivr_edges`, `ivr_prompts`).
2. Validates the tree: no cycles (DFS), all prompts exist in S3, all `to_node` references resolve, all in-group targets exist in `ingroups`.
3. Generates the dialplan XML for the `default` context (one extension per node, one branch-dispatcher per node).
4. Writes to `freeswitch/conf/dialplan/default/70_ivr_{id}.xml`.
5. Also writes/updates the `public` context DID entry file `dialplan/public/10_did_{e164}.xml`.
6. Issues `bgapi reloadxml` via T01 ESL connection.

### 11.2 Generator correctness invariants

- Every node generates exactly two extensions: `ivr_{id}_n{node_id}` (prompt + collect) and `ivr_{id}_n{node_id}_branch` (condition dispatch).
- Terminal action nodes (route_to_ingroup, hangup, voicemail, external_transfer, callback_offer) generate a single extension with no `read` app.
- The entry node id is stored on the `ivrs` row and referenced by the DID entry point.
- A "timeout" edge (special `on_input=''`) and "invalid" edge (special `on_input='*'`) are mandatory per non-terminal node. The renderer rejects saves where these are missing.
- Maximum depth is validated at save time: DFS from entry node counts maximum depth; if > 3 (Phase 1 cap), save is rejected with a specific error message: "Phase 1 IVR trees are limited to 3 levels deep. Upgrade to Phase 2 for deeper trees."

### 11.3 Reload safety

The renderer writes to a temporary file `70_ivr_{id}.xml.tmp`, fsync, then rename (atomic on POSIX). This prevents FS from reading a partially-written file if it happens to reload between write and close.

## 12. Recording consent for inbound calls

### 12.1 Inbound consent vs outbound

C02 PLAN §1.2 explicitly states: "Recording consent (in 2-party-consent states) before agent bridge." This applies to inbound calls too — if an agent in CA receives an inbound call, CA Penal Code §632 still requires disclosure.

**Vici2 approach:** `did_numbers.recording_disclosure_audio` (added by I01 PLAN §3.5) is played at the DID entry point BEFORE the IVR tree begins. This satisfies the disclosure requirement for all calls to that DID regardless of the IVR path taken.

The disclosure is NOT inside the IVR tree — it is played unconditionally at the public context entry extension, before `transfer ivr_{id}_entry`. This matches the I01 PLAN §13 pattern for in-group calls.

### 12.2 Channel vars set at IVR entry

```
vici2_consent_mode     → from did_numbers.consent_mode or tenant default
vici2_consent_status   → 'prompted_assumed' after disclosure plays
consent_record_enabled → 'true' if disclosure played and consent_mode allows recording
```

These are read by R01 when the call eventually reaches an agent conference and recording starts. The IVR itself does not start recording.

## 13. Open questions for PLAN

**Q1: Phase 1 XML file location.** F03 HANDOFF §5.2 reserves `10_*.xml – 89_*.xml` in `public` for I02, and I02 writes DID entry files there. The IVR node extensions belong in the `default` context (not `public`). The naming `70_ivr_{id}.xml` in `default` is chosen to avoid conflicts with other `default` context files. Confirm: is `70_` an acceptable prefix in `default`, or should I02 use a dedicated subdirectory?

**Q2: Renderer location.** The IvrRenderer writes directly to the FS volume. In Phase 2 (multi-FS), it must write to EACH FS host's volume. The multi-FS topology is not yet designed (T01 PLAN §0 bullet 3 says "Phase 1: single FS"). Confirm the Phase 1 assumption: single FS, single volume mount.

**Q3: Phase 2 ivrbridge binary.** Should `ivrbridge` be a subcommand of the existing `dialer/cmd/eslbridge/` binary or a separate `dialer/cmd/ivrbridge/`? T01 PLAN §2 already splits `dialer` and `eslbridge`; a third binary follows the pattern but adds operational complexity.

**Q4: Language selection node spec.** The `lang_select` node type requires playing a bilingual prompt and capturing a digit. Should this be implemented as a regular `collect` node with explicit edges (digit 1 → set lang=en, digit 2 → set lang=es, then continue), or as a special node type with first-class language-setting semantics?

**Q5: IVR traversal log for Phase 1.** The per-call path reconstruction approach (channel vars + HANGUP event) is lossy if channel vars exceed FS's per-channel var storage limit (~4 KB per var). For a 3-level tree with node IDs this is fine. For Phase 2 deeper trees (10+ levels), the path string may approach limits. Confirm Phase 1 cap is sufficient for 3-level × single-digit IDs.

**Q6: DID file naming in public context.** The current proposal writes `10_did_{e164_digits}.xml`. If a DID is `+12125550100`, the file is `10_did_12125550100.xml`. Confirm this doesn't collide with any F03-reserved prefix or any other module's files in the `10_*–89_*` range.
