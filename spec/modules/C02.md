# Module C02 — Recording Consent Handler

| Field | Value |
|---|---|
| Track | Compliance |
| Phase | 1 |
| Effort | 2–3 days |
| Owner agent type | backend-node + telephony |
| Status | NOT_STARTED |

## Goal
Before bridging customer to agent, optionally play a "this call may be recorded" prompt (mandatory in 2-party-consent states by default). If customer rejects via DTMF, hangup without recording. All decisions audited.

## Dependencies
- T04 (originate), R01 (recording), D03 (state lookup)

## Public interface
**Behavior:** in CHANNEL_ANSWER handler (T01), before bridge:
1. Look up `lead.state`.
2. If state ∈ TWO_PARTY_STATES (CA, FL, IL, MD, MA, MT, NH, PA, WA, CT, DE, NV) and campaign.recording_mode != NEVER:
   - `uuid_broadcast <uuid> playback:${recording_consent_audio} aleg`
   - `play_and_get_digits` — '1' to consent, '2' to opt out
3. On consent → bridge + record_session.
4. On opt-out → bridge but no recording (or if campaign requires recording, hang up).

2. Lead in NY (1-party): no prompt; bridges + records normally.
3. Audit log of consent decision per call.

## Acceptance criteria
- [ ] State-based prompt logic.
- [ ] DTMF capture works.
- [ ] Audit per decision.

## Risks
- **Latency added by prompt** (~3-5s). Acceptable for compliance.
- **State unknown for new leads** — default to most restrictive (play prompt).
